🎯 Simulador de Problemas de Rendimiento

Una Azure App Service educativa que Crea intencionadamente problemas de rendimiento para fines de aprendizaje y demostración.

Propósito

Esta aplicación está diseñada para ayudar a Azure App Service Ingenieros de Soporte y Asesores Técnicos:

📖
Aprende Cómo diagnosticar problemas comunes de rendimiento en Azure App Service
🔧
Práctica Uso de Azure herramientas de diagnóstico y monitorización
🎭
Demostrar Anti-patrones de rendimiento en un entorno controlado
👥
Tren Apoyar a los equipos en la identificación y resolución de problemas de rendimiento

⚠️ Advertencia: ¡Esta aplicación crea intencionadamente problemas de rendimiento!

  • 🔥 CPU estrés - Crea hilos dedicados que ejecutan bucles de spin para consumir todos CPU núcleos
  • 📊 Presión de memoria - Asigna y pina bloques de memoria para aumentar el conjunto de trabajo
  • 🧵 Escasez de pozas de hilo - Utiliza sync-over-async anti-patrones para bloquear hilos de pool de hilos
  • 🐌 Solicitudes lentas - Genera solicitudes de larga duración con patrones de sync-over-async para análisis de CLR perfiladores
  • 💥 Caídas de aplicaciones - Provoca fallos mortales para pruebas Azure monitorización de fallos y volcados de memoria
  • ❌ Solicitudes fallidas - Genera HTTP errores 5xx para probar AppLens y Application Insights

Solo despliega esta aplicación en entornos aislados y no de producción.

📊 Salpicadero

La aplicación incluye un panel de control en tiempo real en la URL raíz que muestra:

  • CPU Uso - Utilización actual del procesador
  • Memoria- Tamaños de conjunto de trabajo y heap GC
  • Conjunto de hilos - Hilos activos y longitud de la cola
  • Latencia de petición - Tiempo de respuesta en tiempo real de la sonda (muestra el impacto de la falta de pool de hilos)
  • Simulaciones activas - Actualmente ejecutando simulaciones de problemas

El panel utiliza SignalR para actualizaciones en tiempo real e incluye controles para activar cada tipo de simulación.

Indicadores métricos de color

Las fichas de métricas de CPU y Memoria utilizan codificación dinámica por colores para indicar los niveles de utilización de recursos:

Color Utilización Estado
Negro (por defecto) 0-60% Normal - los recursos están dentro del rango aceptable
Amarillo 60-80% Advertencia: uso elevado, niveles preocupantes
Rojo >80% Peligro: alto uso, posible agotamiento de recursos

Nota: Los umbrales de memoria se calculan dinámicamente en función de la memoria total disponible real reportada por el servidor, no en un valor codificado en fija. Esto garantiza advertencias precisas independientemente de la configuración de la RAM de la máquina.

IDs de registro de eventos y simulación

El Registro de Eventos en la parte inferior del panel registra toda la actividad de simulación con entradas codificadas por colores que coinciden con cada tipo de simulación. Los mensajes relacionados con una instancia de simulación específica se marcan con un subrayado punteado.

Visualización y copia de identificadores de simulación: Pasa el cursor sobre cualquier mensaje subrayado para revelar el mensaje completo ID de simulación (GUID) en una descripción emergente. Haz clic en el mensaje para copiar el ID de simulación en tu portapapeles — el mensaje resaltará brevemente verde para confirmar la copia. Este ID identifica de forma única cada instancia de simulación.

Uso de identificadores de simulación para la resolución de problemas:

  • El mismo ID de simulación se registra en el lado del servidor y aparece en Azure Application Insights y Log Analytics
  • Utiliza este ID para correlacionar eventos del panel con los registros del servidor al analizar problemas de rendimiento
  • Filtra Azure registros por ID de simulación para aislar la telemetría de una simulación específica
  • Véase el Azure Guía de Monitorización para técnicas detalladas de correlación

⏱️ Monitor de latencia de solicitudes

El salpicadero incluye un Monitor de latencia de solicitudes Eso demuestra cómo la falta de pool de hilos afecta al procesamiento real de peticiones. Los valores de latencia mostrados incluyen tanto tiempo de cola como tiempo de procesamiento.

Cómo funciona

  • Un hilo dedicado en segundo plano (que no pertenece al pool de hilos) explora continuamente /api/health/probe
  • La latencia se mide de extremo a extremo: solicitud enviada → respuesta recibida
  • Los resultados se transmiten vía SignalR al panel de control en tiempo real

Lo que observarás

Escenario Latencia (total) Estado Explicación
Funcionamiento normal < 150ms 🟢 Bien Hilos de pool de hilos disponibles
Inanición leve 150ms - 1s 🟡 Degradado Peticiones en cola esperando hilos
Hambre severa > 1s 🔴 Severo Retraso significativo en la cola
Tiempo muerto Años 30 🔴 Crítica No se desbloqueó ningún hilo en el tiempo de espera

Por qué esto importa

Durante la falta de pool de hilos, las métricas de CPU y memoria suelen parecer normales, pero los usuarios experimentan una latencia severa. El monitor de latencia hace que este problema sea invisible visible — puedes ver cómo los tiempos de respuesta aumentan de milisegundos a segundos al activar la simulación sync-over-async .

🔥 CPU Simulación de tensiones

La simulación de CPU Tensiones genera un alto uso de CPU mediante bucles de giro paralelos, saturando todos los núcleos de CPU disponibles durante un periodo determinado.

Cómo funciona

La simulación genera un hilo por cada núcleo CPU , cada uno ejecutando un bucle de giro de espera ocupada (incrementando un contador en un formato cerrado while(true) bucle). Esto consume el 100% de CPU capacidad en todos los núcleos.

Controles del salpicadero

  • Duración: Cuánto tiempo debe durar la prueba de estrés (10-300 segundos, por defecto 30)
  • Inicio/Parada: Iniciar o cancelar la simulación

Lo que observarás

Métrica Durante la simulación Explicación
CPU % ~100% Todos los núcleos están saturados por bucles de espín
Memoria Sin cambios Los bucles de spin no asignan memoria
Latencia Puede aumentar Menos CPU disponible para el procesamiento de solicitudes

Problemas CPU en el mundo real

La alta CPU en la producción suele estar causada por:

  • Algoritmos ineficientes (O(n²) cuando O(n) es posible)
  • Bucles infinitos debido a errores
  • Retroceso excesivo de expresión regular
  • Consultas LINQ no optimizadas con grandes conjuntos de datos
  • Espera ocupada en vez de usar async/await

Herramientas de diagnóstico

  • dotnet-counters monitor -p {PID} --counters System.Runtime
  • dotnet-trace collect -p {PID} para CPU perfilado
  • Azure App Service → Diagnosticar y Resolver Problemas → CPU Uso
  • Application Insights Hoja de rendimiento

💾 Simulación de presión de memoria

La simulación de Presión de Memoria asigna y mantiene grandes matrices de bytes para demostrar patrones de consumo de memoria y comportamiento de recogida de basura.

Cómo funciona

Se asignan grandes arrays de bytes (mínimo 10MB, por defecto 100MB) usando GC.AllocateArray<byte>(pinned: true). La asignación fijada impide que el recolector de basura mueva la memoria, simulando patrones problemáticos de asignación.

Controles del salpicadero

  • Tamaño (MB): Cantidad de memoria a asignar (10-500MB)
  • Asignar: Añade un nuevo bloque de memoria
  • Publicar todo: Libera todos los bloques y disparos asignados GC

Lo que observarás

Métrica Después de la asignación Explicación
Memoria (MB) Aumentos por tamaño de asignación Los gráficos reflejan el aumento del conjunto de trabajo
CPU Breve pico Asignación de memoria y cero
Tras el lanzamiento Caídas de memoria GC. Collect() recupera la memoria

Gran Montón de Objetos (LOH)

Los objetos mayores que 85KB se asignan al Gran Montón de Objetos:

  • LOH se recopila con menos frecuencia (solo colecciones de la Generación 2)
  • Las asignaciones fijas evitan la compactación de memoria
  • Puede provocar fragmentación de la memoria con el tiempo
  • Nuestras asignaciones (10MB+) siempre van a LOH

Problemas de memoria en el mundo real

  • Colecciones estáticas que acumulan datos sin límites
  • No se den de baja los gestores de eventos
  • Implementación IDisposable incorrecta
  • Almacenamiento en caché sin límites de tamaño ni caducidad
  • Mantener referencias a objetos grandes más tiempo del necesario

Herramientas de diagnóstico

  • dotnet-dump collect -p {PID} para volcados de memoria
  • dotnet-gcdump collect -p {PID} para GC análisis de heap
  • Visual Studio Memory Profiler
  • Azure App Service → hoja del conjunto de trabajo de memoria

🧵 Escaseza de pozas de hilo

La simulación de Starvation del Thread Pool demuestra el peligroso anti-patrón "sync-over-async", donde el código síncrono bloquea esperando a que se completen las operaciones asíncronas.

Cómo funciona

La simulación genera múltiples tareas paralelas (por defecto: 2× CPU núcleos) que cada una llama Task.Delay().Wait(). Esto bloquea los hilos del grupo de hilos, provocando:

  1. Los hilos del pool de hilos se bloquean esperando
  2. .NET añade nuevos hilos poco a poco (~1 por segundo)
  3. Las solicitudes entrantes se colocan esperando hilos
  4. La latencia de las peticiones aumenta drásticamente

Controles del salpicadero

  • Duración: Cuánto dura cada operación de bloqueo (por defecto 15 segundos)
  • Hilos: Número de hilos bloqueadores que aparecen
  • Inicio/Parada: Iniciar o cancelar la simulación

El anti-patrón Sincronización sobre Asíncrona

Estos patrones bloquean los hilos y deben evitarse:

// ❌ BAD - Blocks thread waiting for Task
var result = SomeAsyncMethod().Result;
var result = SomeAsyncMethod().GetAwaiter().GetResult();
SomeAsyncMethod().Wait();

// ✅ GOOD - Properly awaits the Task
var result = await SomeAsyncMethod();

Lo que observarás

Métrica Durante la inanición Explicación
CPU BAJO Los hilos están esperando, no funcionan
Memoria Normal No se produce ninguna fuga de memoria
Recuento de hilos Ascenso lento .NET añade ~1 hilo por segundo
Latencia 🔴 De picos a segundos Peticiones en cola esperando hilos

Por qué CPU es baja pero la latencia alta

Esta es la señal de la falta de hilos. El CPU no funciona porque todos los hilos están bloqueados en espera. No pueden iniciar nuevas solicitudes porque no hay hilos disponibles. Parece que la app está inactiva, pero los usuarios experimentan tiempos de espera.

Causas reales

  • Llamar a bibliotecas asíncronas desde código de sincronización: httpClient.GetAsync(url).Result
  • Mezcla de sincronización y asincrónico en constructores
  • Usando .Result en propiedades
  • Llamando a métodos asíncronos desde Dispose()
  • Bibliotecas de terceros que bloquean internamente

Cómo arreglar

  • Uso await Hasta arriba en la pila de llamadas ("asíncrono hasta abajo")
  • Evita .Result, .Wait(), y .GetAwaiter().GetResult()
  • Uso ConfigureAwait(false) En el código de la biblioteca
  • Considera IAsyncEnumerable para datos en streaming

Herramientas de diagnóstico

  • dotnet-counters: Mira "ThreadPool Thread Count" y "ThreadPool Queue Length"
  • Los tiempos de respuesta aumentan para TODOS los puntos finales simultáneamente
  • Azure App Service → Diagnosticar y Resolver Problemas → Número de Hilos

🐌 Simulador de Peticiones Lentas

El Simulador de Peticiones Lentas genera peticiones de larga duración (~25 segundos cada una) utilizando patrones de sync-over-async diseñados específicamente para CLR Formación de perfilador.

Propósito

A diferencia de la simulación de Starvation del Pool de Hilos (que provoca muchos bloques cortos), esta crea solicitudes largas individuales que son claramente visibles en las pilas de llamadas al perfilador. Los nombres de los métodos son intencionadamente descriptivos para facilitar la identificación de los patrones de bloqueo.

Tres escenarios de sincronización sobre asincronismo

Escenario Patrón de método Qué buscar
Sencillo _BLOCKING_HERE Task.Delay().Wait() Provocando bloqueo
Anidado _BLOCKS_INTERNALLY Métodos de sincronización que bloquean internamente
Base de datos _SYNC_BLOCK GetAwaiter().GetResult() Patrón

Configuraciones recomendadas

Para un trazado de CLR Profiler de 60 segundos:

  • Tiempo de ejecución de la solicitud: 25 segundos (solo tiempo de procesamiento, excluyendo el tiempo de cola)
  • Intervalo: 10 segundos (múltiples muestras)
  • Solicitudes máximas: 6 (no abruma)

Nota sobre el tiempo de espera: Si el pool de hilos está sin recursos, puedes ver un gran hueco en la traza antes de que empiece la solicitud. Esto es "Tiempo de cola" (esperar un hilo) frente a "Tiempo de ejecución" (ejecutar el código).

Usar con CLR Profiler

  1. Iniciar un trazado CPU Profiler (60-120 segundos)
  2. Clic 🐌 Iniciar solicitudes lentas en el salpicadero
  3. Esperar a que el rastreo se complete
  4. Descargar y analizar: busca los nombres descriptivos de los métodos

Véase el Azure Guía de diagnóstico Para instrucciones detalladas de análisis del perfilador.

💥 Simulación de accidentes

El salpicadero incluye un Simulación de accidentes Función para probar Azure monitorización de fallos y recogida de volcado de memoria.

⚠️ Advertencia: ¡Esto CANCELARÁ la solicitud!

Las simulaciones de accidentes terminan inmediatamente el proceso. Azure App Service reiniciará automáticamente la aplicación tras un fallo. Asegúrate de tener activado Azure Monitorización de Bloqueos antes de activar un bloqueo.

Tipos de accidentes disponibles

Tipo Descripción Mejor para
FailFast Llamadas Environment.FailFast() que termina inmediatamente con un Informe de Error de Windows Pruebas generales de choque (recomendable)
StackOverflow Recursión infinita que causa desbordamiento de pila - no se puede detectar Práctica de análisis de trazas de pila
UnhandledException Lanza una excepción no gestionada en el hilo de segundo plano Educación en manejo de excepciones
ViolaciónDeacceso Escribe en memoria inválida (puntero nulo) Depuración de código nativo/inseguro
FueraMemoria Asigna memoria hasta que el proceso se bloquea Análisis de volcado de memoria

Azure Configuración de Monitorización de Fallos

  1. En Azure Portal, ve a tu App Service
  2. Navegar a Diagnosticar y resolver problemas
  3. Búsqueda de "Monitorización de caídas"
  4. Activar la monitorización de fallos con "Recoger volcado de memoria"
  5. Provoca un bloqueo usando el salpicadero o API
  6. Descarga el volcado de fallos desde el portal Azure
  7. Analizar con Visual Studio o WinDbg

❌ Simulación de solicitudes fallidas

El salpicadero incluye un Solicitudes fallidas función para generar HTTP errores 5xx para probar Azure herramientas de diagnóstico como AppLens y Application Insights.

Propósito

Esta simulación genera HTTP 500 respuestas de error interno del servidor que son visibles en:

  • AppLens - Azure Portal → Diagnosticar y resolver problemas
  • Application Insights - Hoja de fallos
  • Azure Monitor - Alertas de tasa de error

Cómo funciona

  1. Genera un hilo en segundo plano para generar peticiones
  2. Cada solicitud llama /api/loadtest con probabilidad de error del 100%
  3. El endpoint de la prueba de carga genera un tipo de excepción aleatorio
  4. Cada solicitud tarda ~1,5 segundos en visibilidad de latencia

Tipos de excepción

La simulación lanza aleatoriamente uno de estos tipos de excepción por solicitud:

Tipo de excepción Descripción
NullReferenceException Referencia de objeto no asignada a una instancia
TimeoutException Operación agotada
InvalidOperationException Operación no válida para el estado actual
ArgumentException Valor fuera del rango esperado
HttpRequestException Error al enviar HTTP solicitud
DivideByZeroException Intentó dividir por cero
KeyNotFoundException Clave no presente en el diccionario
IndexOutOfRangeException Índice fuera de los límites del array

Registro de eventos del Panel

Cada solicitud fallida aparece en el Registro de Eventos con:

  • Color marrón - Estilo distintivo para la visibilidad
  • ❌ Icono - Indicador de solicitud fallida
  • Tipo de excepción - La excepción específica que se lanzó
  • Latencia - Duración de la solicitud en milisegundos

Véase el Azure Guía de diagnóstico Para instrucciones detalladas de análisis AppLens .

⚙️ Configuración opcional

Puedes personalizar el comportamiento de la aplicación usando Azure App Service variables de entorno.

Tasa de Prueba de Salud

Controla con qué frecuencia el servidor envía sondas de salud para medir la latencia de las solicitudes. Todas las sondas se enrutan a través del frontend Azure para capturar latencias realistas de extremo a extremo.

Variable de entorno

Nombre de la variable Descripción Default
HEALTH_PROBE_RATE Intervalo de sonda en milisegundos. Mínimo 100ms. 200

Configuración mediante Azure CLI

# Ralentiza las sondas si CLR perfilador muestra solicitudes superpuestasaz webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \ --ajustes HEALTH_PROBE_RATE=400

Tiempo de espera en reposo

Cuando la aplicación está inactiva (sin conexiones al panel ni solicitudes de pruebas de carga), las sondas de salud se suspenden automáticamente para reducir el tráfico de red innecesario al frontend de Azurey a la telemetría Application Insights .

Variable de entorno

Nombre de la variable Descripción Default
IDLE_TIMEOUT_MINUTES Minutos de inactividad antes de suspender las sondas de salud. La actividad se reanuda automáticamente cuando se abre el panel de control o se recibe cualquier solicitud. 20

Configuración mediante Azure CLI

# Extiende el tiempo muerto en reposo a 30 minutosaz webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \ --ajustes IDLE_TIMEOUT_MINUTES=30

Pie de página personalizado

Establece un mensaje personalizado en el pie de página que aparezca en todas las páginas. El pie de página soporta HTML, lo que permite incluir enlaces.

Variable de entorno

Nombre de la variable Descripción
PAGE_FOOTER Contenido HTML para la segunda línea del pie de página. Si no está activado, solo se muestra la descripción de la app y la información de la build.

Valor de ejemplo

Creado por <a href="https://yoursite.com">Your Team</a> con fines de entrenamiento

Configuración mediante Azure CLI

# Coloca el pie de página mediante Azure CLIaz webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \ --ajustes 'PAGE_FOOTER=Creado por <a href="https://yoursite.com">Tu equipo</a> para fines de entrenamiento'

Ambientación a través de Azure Portal

  1. Navega hasta tu App Service en el Portal Azure
  2. Ir a Escenarios → Variables de entorno
  3. Clic + Añadir
  4. Nombre del conjunto: PAGE_FOOTER
  5. Valor fijado: Contenido del pie de página HTML
  6. Clic Solicita, entonces Confirmar

GitHub Enlace al Repositorio

Muestra un enlace al repositorio de GitHub en el menú lateral. El enlace está oculto por defecto y solo aparece cuando ambas variables de entorno están configuradas.

Variables ambientales

Nombre de la variable Descripción
GITHUB_USER_NAME GitHub nombre de usuario u organización (por ejemplo, contoso)
GITHUB_REPO_NAME Nombre del repositorio (por ejemplo, perf-simulator)

Configuración mediante Azure CLI

# Activar GitHub enlace en el menú lateralaz webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \ --settings GITHUB_USER_NAME=contoso GITHUB_REPO_NAME=simulador de perfs

🌐 Selección de idioma (Traducción de la interfaz)

La interfaz del panel de control puede mostrarse en cualquier idioma soportado por Azure Traductor de Servicios Cognitivos. Las traducciones se generan automáticamente al arrancar usando IA y se almacenan en caché para las solicitudes posteriores.

⚠️ Aviso de traducción de IA: Las traducciones son generadas por Azure Traductor de Servicios Cognitivos (traducción automática) y pueden no ser completamente precisas. Los términos técnicos (por ejemplo, CPU, SignalR, ThreadPool) se conservan intencionadamente en inglés.

Cómo funciona

  1. Establece el UI_LANGUAGE variable de entorno a una Código de idioma ISO 639-1 (por ejemplo, es para español, fr para el francés, ja para japonés)
  2. Al iniciar, la app comprueba si existe un archivo de localización traducido para ese idioma
  3. Si no (o si las cadenas fuente en inglés han cambiado), llama a Azure traductor para generar una
  4. El archivo traducido se almacena en caché — los arranques posteriores utilizan la caché sin coste de traducción
  5. El panel carga las cadenas traducidas y las aplica a la interfaz

Variables ambientales

Nombre de la variable Descripción Default
UI_LANGUAGE Código de idioma ISO 639-1 para la interfaz del panel de control. A ajustar en para el inglés (no se necesita traducción). en
TRANSLATOR_API_KEY Azure clave de suscripción a Cognitive Services Translator. Obligatorio para idiomas que no son inglés. (no fijada)
TRANSLATOR_ENDPOINT Traductor API URL del endpoint. https://api.cognitive.microsofttranslator.com
TRANSLATOR_REGION Azure región del recurso Traductor. eastus

# Pon la interfaz en españolaz webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \ --ajustes UI_LANGUAGE=es TRANSLATOR_API_KEY=tu-tecla TRANSLATOR_REGION=eastus

Ambientación a través de Azure Portal

  1. Navega hasta tu App Service en el Portal Azure
  2. Ir a Escenarios → Variables de entorno
  3. Clic + Añadir y nombre del conjunto: UI_LANGUAGE, Valor: tu código de lenguaje (por ejemplo, es)
  4. Clic + Añadir y nombre del conjunto: TRANSLATOR_API_KEY, Valor: tu Azure Clave traductora
  5. Clic + Añadir y nombre del conjunto: TRANSLATOR_REGION, Valor: tu región de recursos traductora
  6. Clic Solicita, entonces Confirmar (se requiere reinicio)
