🎯 Performance Problem Simulator

An educational Azure App Service application that intentionally creates performance problems for learning and demonstration purposes.

Purpose

This application is designed to help Azure App Service Support engineers and Technical Advisors:

📖
Learn How to diagnose common performance issues in Azure App Service
🔧
Practice Using Azure diagnostics and monitoring tools
🎭
Demonstrate Performance anti-patterns in a controlled environment
👥
Train Support teams on identifying and resolving performance problems

⚠️ Warning - This application intentionally creates performance problems!

  • 🔥 CPU stress - Creates dedicated threads running spin loops to consume all CPU cores
  • 📊 Memory pressure - Allocates and pins memory blocks to increase working set
  • 🧵 Thread pool starvation - Uses sync-over-async anti-patterns to block thread pool threads
  • 🐌 Slow requests - Generates long-running requests with sync-over-async patterns for CLR Profiler analysis
  • 💥 Application crashes - Triggers fatal crashes for testing Azure Crash Monitoring and memory dumps
  • ❌ Failed requests - Generates HTTP 5xx errors for testing AppLens and Application Insights

Only deploy this application in isolated, non-production environments.

📊 Dashboard

The application includes a real-time dashboard at the root URL that shows:

  • CPU usage - Current processor utilization
  • Memory - Working set and GC heap sizes
  • Thread pool - Active threads and queue length
  • Request latency - Real-time probe response time (shows impact of thread pool starvation)
  • Active simulations - Currently running problem simulations

The dashboard uses SignalR for real-time updates and includes controls to trigger each type of simulation.

Metric Color Indicators

The CPU and Memory metric tiles use dynamic color coding to indicate resource utilization levels:

Color Utilization Status
Black (default) 0-60% Normal - resources are within acceptable range
Yellow 60-80% Warning - elevated usage, approaching concerning levels
Red >80% Danger - high usage, potential resource exhaustion

Note: Memory thresholds are calculated dynamically based on the actual total available memory reported by the server, not a hardcoded value. This ensures accurate warnings regardless of the machine's RAM configuration.

Event Log & Simulation IDs

The Event Log at the bottom of the dashboard records all simulation activity with color-coded entries matching each simulation type. Messages related to a specific simulation instance are marked with a dotted underline.

Viewing Simulation IDs: Hover over any underlined message to reveal the full Simulation ID (GUID) in a tooltip. This ID uniquely identifies each simulation instance.

Using Simulation IDs for Troubleshooting:

  • The same Simulation ID is logged server-side and appears in Azure Application Insights and Log Analytics
  • Use this ID to correlate dashboard events with server logs when analyzing performance issues
  • Filter Azure logs by Simulation ID to isolate telemetry from a specific simulation run
  • See the Azure Monitoring Guide for detailed correlation techniques

⏱️ Request Latency Monitor

The dashboard includes a Request Latency Monitor that demonstrates how thread pool starvation affects real-world request processing. The latency values shown include both queue time and processing time.

How It Works

  • A dedicated background thread (not from the thread pool) continuously probes /api/health/probe
  • Latency is measured end-to-end: request sent → response received
  • Results are broadcast via SignalR to the dashboard in real-time

What You'll Observe

Scenario Latency (Total) Status Explanation
Normal operation < 150ms 🟢 Good Thread pool threads available
Mild starvation 150ms - 1s 🟡 Degraded Requests queued waiting for threads
Severe starvation > 1s 🔴 Severe Significant queuing delay
Timeout 30s 🔴 Critical No thread became available within timeout

Why This Matters

During thread pool starvation, CPU and memory metrics often look normal, but users experience severe latency. The latency monitor makes this invisible problem visible — you can watch response times spike from milliseconds to seconds when triggering the sync-over-async simulation.

🔥 CPU Stress Simulation

The CPU Stress simulation creates high CPU usage through parallel spin loops, saturating all available CPU cores for a specified duration.

How It Works

The simulation spawns one thread per CPU core, each running a busy-wait spin loop (incrementing a counter in a tight while(true) loop). This consumes 100% of CPU capacity across all cores.

Dashboard Controls

  • Duration: How long to run the stress test (10-300 seconds, default 30)
  • Start/Stop: Begin or cancel the simulation

What You'll Observe

Metric During Simulation Explanation
CPU % ~100% All cores saturated by spin loops
Memory Unchanged Spin loops don't allocate memory
Latency May increase Less CPU available for request processing

Real-World CPU Issues

High CPU in production is typically caused by:

  • Inefficient algorithms (O(n²) when O(n) is possible)
  • Infinite loops due to bugs
  • Excessive regular expression backtracking
  • Unoptimized LINQ queries with large datasets
  • Busy-waiting instead of using async/await

Diagnosis Tools

  • dotnet-counters monitor -p {PID} --counters System.Runtime
  • dotnet-trace collect -p {PID} for CPU profiling
  • Azure App Service → Diagnose and Solve Problems → CPU usage
  • Application Insights Performance blade

💾 Memory Pressure Simulation

The Memory Pressure simulation allocates and holds large byte arrays to demonstrate memory consumption patterns and garbage collection behavior.

How It Works

Large byte arrays (minimum 10MB, default 100MB) are allocated using GC.AllocateArray<byte>(pinned: true). The pinned allocation prevents the garbage collector from moving the memory, simulating problematic allocation patterns.

Dashboard Controls

  • Size (MB): Amount of memory to allocate (10-500MB)
  • Allocate: Adds a new memory block
  • Release All: Frees all allocated blocks and triggers GC

What You'll Observe

Metric After Allocation Explanation
Memory (MB) Increases by allocation size Charts reflect working set increase
CPU Brief spike Memory allocation and zeroing
After Release Memory drops GC.Collect() reclaims memory

Large Object Heap (LOH)

Objects larger than 85KB are allocated on the Large Object Heap:

  • LOH is collected less frequently (Gen 2 collections only)
  • Pinned allocations prevent memory compaction
  • Can lead to memory fragmentation over time
  • Our allocations (10MB+) always go to LOH

Real-World Memory Issues

  • Static collections that accumulate data without limits
  • Event handlers not being unsubscribed
  • Improper IDisposable implementation
  • Caching without size limits or expiration
  • Keeping references to large objects longer than needed

Diagnosis Tools

  • dotnet-dump collect -p {PID} for memory dumps
  • dotnet-gcdump collect -p {PID} for GC heap analysis
  • Visual Studio Memory Profiler
  • Azure App Service → Memory Working Set blade

🧵 Thread Pool Starvation

The Thread Pool Starvation simulation demonstrates the dangerous "sync-over-async" anti-pattern, where synchronous code blocks waiting for asynchronous operations to complete.

How It Works

The simulation spawns multiple parallel tasks (default: 2× CPU cores) that each call Task.Delay().Wait(). This blocks thread pool threads, causing:

  1. Thread pool threads become blocked waiting
  2. .NET slowly adds new threads (~1 per second)
  3. Incoming requests queue up waiting for threads
  4. Request latency spikes dramatically

Dashboard Controls

  • Duration: How long each blocking operation lasts (default 15s)
  • Threads: Number of blocking threads to spawn
  • Start/Stop: Begin or cancel the simulation

The Sync-Over-Async Anti-Pattern

These patterns block threads and should be avoided:

// ❌ BAD - Blocks thread waiting for Task
var result = SomeAsyncMethod().Result;
var result = SomeAsyncMethod().GetAwaiter().GetResult();
SomeAsyncMethod().Wait();

// ✅ GOOD - Properly awaits the Task
var result = await SomeAsyncMethod();

What You'll Observe

Metric During Starvation Explanation
CPU LOW Threads are waiting, not working
Memory Normal No memory leak occurring
Thread Count Slowly climbing .NET adds ~1 thread/second
Latency 🔴 Spikes to seconds Requests queued waiting for threads

Why CPU is Low but Latency is High

This is the signature of thread pool starvation. The CPU isn't working because all threads are blocked waiting. New requests can't start because there are no available threads. It looks like the app is idle, but users experience timeouts.

Real-World Causes

  • Calling async libraries from sync code: httpClient.GetAsync(url).Result
  • Mixing sync and async in constructors
  • Using .Result in properties
  • Calling async methods from Dispose()
  • Third-party libraries that block internally

How to Fix

  • Use await all the way up the call stack ("async all the way down")
  • Avoid .Result, .Wait(), and .GetAwaiter().GetResult()
  • Use ConfigureAwait(false) in library code
  • Consider IAsyncEnumerable for streaming data

Diagnosis Tools

  • dotnet-counters: Watch "ThreadPool Thread Count" and "ThreadPool Queue Length"
  • Response times spike for ALL endpoints simultaneously
  • Azure App Service → Diagnose and Solve Problems → Thread count

🐌 Slow Request Simulator

The Slow Request Simulator generates long-running requests (~25 seconds each) using sync-over-async patterns specifically designed for CLR Profiler training.

Purpose

Unlike the Thread Pool Starvation simulation (which causes many short blocks), this creates individual long requests that are clearly visible in profiler call stacks. The method names are intentionally descriptive to make identifying the blocking patterns easy.

Three Sync-Over-Async Scenarios

Scenario Method Pattern What to Look For
Simple _BLOCKING_HERE Task.Delay().Wait() causing blocking
Nested _BLOCKS_INTERNALLY Sync methods that block internally
Database _SYNC_BLOCK GetAwaiter().GetResult() pattern

Recommended Settings

For a 60-second CLR Profiler trace:

  • Request Execution Time: 25 seconds (processing time only, excludes queue time)
  • Interval: 10 seconds (multiple samples)
  • Max Requests: 6 (doesn't overwhelm)

Note on Queue Time: If the thread pool is starved, you may see a large gap in the trace before the request starts. This is "Queue Time" (waiting for a thread) vs "Execution Time" (running the code).

Using with CLR Profiler

  1. Start a CPU Profiler trace (60-120 seconds)
  2. Click 🐌 Start Slow Requests in the dashboard
  3. Wait for trace to complete
  4. Download and analyze - look for the descriptive method names

See the Azure Diagnostics Guide for detailed profiler analysis instructions.

💥 Crash Simulation

The dashboard includes a Crash Simulation feature for testing Azure crash monitoring and memory dump collection.

⚠️ Warning - This will TERMINATE the application!

Crash simulations immediately terminate the process. Azure App Service will automatically restart the application after a crash. Make sure you have Azure Crash Monitoring enabled before triggering a crash.

Available Crash Types

Type Description Best For
FailFast Calls Environment.FailFast() which immediately terminates with a Windows Error Report General crash testing (recommended)
StackOverflow Infinite recursion causing stack overflow - cannot be caught Stack trace analysis practice
UnhandledException Throws unhandled exception on background thread Exception handling education
AccessViolation Writes to invalid memory (null pointer) Native/unsafe code debugging
OutOfMemory Allocates memory until the process crashes Memory dump analysis

Azure Crash Monitoring Setup

  1. In Azure Portal, go to your App Service
  2. Navigate to Diagnose and solve problems
  3. Search for "Crash Monitoring"
  4. Enable crash monitoring with "Collect Memory Dump"
  5. Trigger a crash using the dashboard or API
  6. Download the crash dump from the Azure portal
  7. Analyze with Visual Studio or WinDbg

❌ Failed Requests Simulation

The dashboard includes a Failed Requests feature for generating HTTP 5xx errors to test Azure diagnostics tools like AppLens and Application Insights.

Purpose

This simulation generates HTTP 500 Internal Server Error responses that are visible in:

  • AppLens - Azure Portal → Diagnose and solve problems
  • Application Insights - Failures blade
  • Azure Monitor - Error rate alerts

How It Works

  1. Spawns a background thread to generate requests
  2. Each request calls /api/loadtest with 100% error probability
  3. The load test endpoint throws a random exception type
  4. Each request takes ~1.5 seconds for latency visibility

Exception Types

The simulation randomly throws one of these exception types per request:

Exception Type Description
NullReferenceException Object reference not set to an instance
TimeoutException Operation timed out
InvalidOperationException Operation not valid for current state
ArgumentException Value out of expected range
HttpRequestException Error sending HTTP request
DivideByZeroException Attempted to divide by zero
KeyNotFoundException Key not present in dictionary
IndexOutOfRangeException Index outside array bounds

Dashboard Event Log

Each failed request appears in the Event Log with:

  • Brown color - Distinctive styling for visibility
  • ❌ icon - Failed request indicator
  • Exception type - The specific exception that was thrown
  • Latency - Request duration in milliseconds

See the Azure Diagnostics Guide for detailed AppLens analysis instructions.
