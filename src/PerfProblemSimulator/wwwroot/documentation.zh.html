API 性能问题模拟器的参考、指南和使用示例

🎯 性能问题模拟器

一个教育 Azure App Service 应用 故意制造性能问题 用于学习和演示目的。

目的

该应用程序旨在帮助 Azure App Service 支持工程师和技术顾问：

📖
学习 如何诊断Azure App Service常见的性能问题
🔧
实践 使用 Azure 诊断和监测工具
🎭
演示 受控环境中的性能反模式
👥
列车 支持团队识别和解决绩效问题

⚠️ 警告——本应用故意制造性能问题！

  • 🔥 CPU 压力 - 创建专用线程运行自旋环，消耗所有 CPU 核心
  • 📊 记忆压力 - 分配并钉住内存块以增加工作集
  • 🧵 线池饥饿 - 使用 sync-over-async 反模式来阻断线程池线程
  • 🐌 慢速请求 - 生成带有 sync-over-async 模式的长期请求，用于 CLR 分析
  • 💥 应用程序崩溃 - 触发致命崩溃以进行测试 Azure 崩溃监控和内存转储
  • ❌ 失败的请求 - 产生 HTTP 个5xx错误用于测试 AppLens 和测试 Application Insights

仅在隔离的非生产环境中部署此应用。

📊 仪表盘

该应用在根URL处包含一个实时仪表盘，显示：

  • CPU 用法 - 当前处理器利用率
  • 记忆 - 工作集和 GC 堆大小
  • 线程池 - 活跃线程和队列长度
  • 请求延迟 - 实时探针响应时间（显示线程池饥饿的影响）
  • 主动仿真 - 目前正在运行问题模拟

仪表盘使用 SignalR 进行实时更新，并包含触发每种模拟类型的控件。

公制颜色指示器

CPU和内存度量图块使用动态颜色编码来表示资源利用率水平：

颜色 利用情况 现状
黑色（默认） 0-60% 正常——资源在可接受范围内
黄色 60-80% 警告——使用率升高，接近令人担忧的水平
红色 >80% 危险——高使用率，潜在资源枯竭

注： 内存阈值是基于服务器报告的实际总可用内存动态计算的，而非硬编码值。这确保了无论机器内存配置如何都能准确发出警告。

事件日志与仿真ID

仪表盘底部的事件日志记录所有仿真活动，并以颜色编码的条目对应各仿真类型。与特定仿真实例相关的消息会用 虚线下划线.

查看和复制仿真ID： 将鼠标悬停在任何下划线的消息上即可显示完整内容 仿真ID （GUID）在提示里。 点击消息，将模拟ID复制到你的剪贴板上 —— 消息会短暂高亮绿色以确认复制。该ID唯一标识每个仿真实例。

使用仿真ID进行故障排除：

  • 同一仿真ID在服务器端记录，并出现在 Azure Application Insights 和Log Analytics中
  • 在分析性能问题时，利用该ID将仪表盘事件与服务器日志关联起来
  • 通过模拟ID过滤 Azure 日志，以将遥测数据与特定仿真运行隔离开来
  • 参见 Azure 监测指南 详细的相关技术

⏱️ 请求延迟监控器

仪表盘包含一个 请求延迟监控器 这展示了线程池饥饿如何影响现实世界的请求处理。显示的延迟值包括 队列时间和处理时间.

工作原理

  • 一个专用的后台线程（非线程池中线程）持续探测 /api/health/probe
  • 延迟是端到端测量的：请求发送→响应
  • 结果通过 SignalR 实时广播到仪表盘

你将观察到的内容

剧情 延迟（总计） 现状 解释
正常运行 < 150毫秒 🟢 很好 可用的线程池线程
轻度饥饿 150毫秒 - 1秒 🟡 退化 等待线程的请求排队
严重饥饿 > 1 🔴 严重 显著的排队延迟
暂停 30多岁 🔴 批判 超时内没有线程出现

为什么这很重要

在线程池饥饿期，显 CPU 和内存指标通常正常，但用户会经历严重的延迟。延迟监控器让这个问题变得隐形 可见 ——当触发 sync-over-async 模拟时，响应时间会从毫秒激增到几秒。

🔥 CPU 应力模拟

CPU应力仿真通过并行自旋环路产生高CPU使用率，使所有可用CPU磁芯在指定时间内饱和。

工作原理

仿真每个核心生成一个线程 CPU ，每个线程运行一个忙等待自旋环（在紧密 while(true) 循环）。这会消耗所有核心的100% CPU 容量。

仪表盘控制

  • 持续时间： 压力测试需要多长时间（10-300秒，默认30秒）
  • 启动/停止： 开始或取消模拟

你将观察到的内容

度规 模拟过程中 解释
CPU % ~100% 所有磁芯都被自旋环饱和
记忆 未变 自旋环不分配内存
延迟 可能会增加 请求处理的可用 CPU 减少

现实世界的 CPU 问题

生产 CPU 高通常由以下原因引起：

  • 低效算法（当O（n）可能时为O（n²）
  • 由于bug导致的无限循环
  • 过度正则表达式回溯
  • 未优化的LINQ查询处理大数据集
  • 忙等待代替异步/等待

诊断工具

  • dotnet-counters monitor -p {PID} --counters System.Runtime
  • dotnet-trace collect -p {PID} 用于 CPU 画像
  • Azure App Service → 诊断和解决使用→ CPU 问题
  • Application Insights 性能刀

💾 内存压力仿真

内存压力模拟分配并保存大字节数组，以展示内存消耗模式和垃圾回收行为。

工作原理

大字节阵列（最小10MB，默认100MB）通过以下方式分配 GC.AllocateArray<byte>(pinned: true).钉置分配防止垃圾回收器移动内存，模拟有问题的分配模式。

仪表盘控制

  • 尺寸（MB）： 需要分配的内存容量（10-500MB）
  • 分配： 新增一个内存块
  • 全部发布： 释放所有分配的方块和触发器 GC

你将观察到的内容

度规 分配后 解释
内存（MB） 按分配规模增加 图表反映了工作集的增加情况
CPU 短暂的峰值 内存分配与归零
发行后 内存丢失 GC。Collect（） 回收内存

大型对象堆（LOH）

大于85KB的对象分配到大型对象堆上：

  • LOH的收集频率较低（仅限第二代收藏）
  • 钉置配置防止内存压缩
  • 随着时间推移，可能导致记忆碎片化
  • 我们的分配（10MB+）总是分配给LOH

现实世界的记忆问题

  • 静态集合，可以无限积累数据
  • 事件处理程序未取消订阅
  • 不当的ID可分割实现
  • 无大小限制或无过期的缓存
  • 对大型对象的引用保持比必要的时间更长

诊断工具

  • dotnet-dump collect -p {PID} 用于内存转储
  • dotnet-gcdump collect -p {PID} 对于 GC 堆分析
  • Visual Studio 内存分析器
  • Azure App Service → 记忆工作组刀片

🧵 线池饥饿

线程池饥饿模拟展示了危险的“sync-over-async”反模式，即同步代码块等待异步操作完成。

工作原理

仿真会生成多个并行任务（默认：2× CPU 个核心），每个任务调用这些任务 Task.Delay().Wait().这会阻断线程池线程，导致：

  1. 线程池线程变成阻塞等待
  2. .NET 缓慢添加新线程（~每秒1条）
  3. 收到的请求排队等待线程
  4. 请求延迟会大幅增加

仪表盘控制

  • 持续时间： 每次阻挡行动持续多久（默认15秒）
  • 讨论串： 需要生成的阻塞线程数量
  • 启动/停止： 开始或取消模拟

同步过非同步反模式

这些图案会阻断线头，应避免：

// ❌ BAD - Blocks thread waiting for Task
var result = SomeAsyncMethod().Result;
var result = SomeAsyncMethod().GetAwaiter().GetResult();
SomeAsyncMethod().Wait();

// ✅ GOOD - Properly awaits the Task
var result = await SomeAsyncMethod();

你将观察到的内容

度规 饥荒时期 解释
CPU 低 线程在等待，但无法正常工作
记忆 普通 没有发生内存泄漏
线数 缓慢攀升 .NET 每秒增加~1个线程
延迟 🔴 秒数激增 等待线程的请求排队

为什么 CPU 低而延迟高

这就是线池饥饿的特征。 CPU 无法工作，因为所有线程都被屏蔽等待。新请求无法启动，因为没有可用的线程。看起来应用处于空闲状态，但用户会遇到超时。

现实世界的原因

  • 从同步代码调用异步库： httpClient.GetAsync(url).Result
  • 构造函数中的同步与异步混合
  • 使用 .Result 在性质中
  • 调用Dispose（）
  • 内部阻挡的第三方库

如何修复

  • 用途 await 呼叫堆栈的最高端（“全速不同步”）
  • 避免 .Result, .Wait()， 和 .GetAwaiter().GetResult()
  • 用途 ConfigureAwait(false) 在图书馆代码中
  • 考虑 IAsyncEnumerable 用于流媒体数据

诊断工具

  • dotnet-counters观看“ThreadPool 线数”和“ThreadPool 队列长度”
  • 所有端点的响应时间同时激增
  • Azure App Service → →线程数诊断和解决问题

🐌 慢速请求模拟器

慢速请求模拟器生成长时间运行的请求（每个请求约25秒），使用专门设计的 sync-over-async 模式 CLR 画像师培训.

目的

与线程池饥饿模拟（导致许多短块）不同，这会创建单个长请求，这些请求在分析器调用栈中清晰可见。方法名称有意描述性，以便识别阻塞模式。

三种同步跨异步场景

剧情 方法模式 注意点
很简单 _BLOCKING_HERE Task.Delay().Wait() 导致阻塞
嵌套 _BLOCKS_INTERNALLY 内部阻挡的同步方法
数据库 _SYNC_BLOCK GetAwaiter().GetResult() 图案

推荐设置

对于60秒的 CLR 分析器追踪：

  • 请求执行时间： 25秒（仅处理时间，不包括排队时间）
  • 中场休息： 10秒（多采样）
  • 最大请求： 6（不会让人感到压力过大）

排队时间说明： 如果线程池缺粮，你可能会在请求开始前看到走线中出现较大的空隙。这是“队列时间”（等待线程）与“执行时间”（运行代码）。

配合 CLR 分析器使用

  1. 开始 CPU 分析器追踪（60-120秒）
  2. 咔哒 🐌 开始慢速请求 在仪表盘中
  3. 等待追踪完成
  4. 下载并分析——寻找描述性方法名称

参见 Azure 诊断指南 详细的剖面分析说明。

💥 撞车模拟

仪表盘包含一个 撞车模拟 用于测试 Azure 崩溃监控和内存转储收集的功能。

⚠️ 警告——这将终止申请！

碰撞模拟会立即终止该进程。 Azure App Service 会在崩溃后自动重启应用。在触发崩溃前，确保你已启用 Azure 崩溃监控。

可用的撞击类型

类型 描述 最佳
FailFast 叫声 Environment.FailFast() 但该报告立即以Windows错误报告结束 一般碰撞测试（推荐）
StackOverflow 无限递归导致堆栈溢出——无法捕捉 栈迹分析实践
未处理异常 在后台线程中抛出未处理的异常 例外处理教育
访问违规 写入无效内存（空指针） 原生/不安全代码调试
记忆之外 分配内存直到进程崩溃 内存转储分析

Azure 崩溃监控设置

  1. 在 Azure 门户，前往你的 App Service
  2. 导航至此 诊断和解决问题
  3. 搜索 “碰撞监测”
  4. 启用碰撞监控 “收集内存翻储”
  5. 通过仪表盘或API触发崩溃
  6. 从 Azure 门户下载崩溃转储文件
  7. 用Visual Studio或WinDbg分析

❌ 失败请求模拟

仪表盘包含一个 未完成的请求 用于生成 HTTP 个5xx错误，用于测试 Azure 诊断工具如 AppLens 和 Application Insights。

目的

该模拟生成 HTTP 500个内部服务器错误响应，具体表现为：

  • AppLens - Azure 门户 → 诊断和解决问题
  • Application Insights - 失效刀片
  • Azure 监视器 - 错误率警报

工作原理

  1. 生成一个后台线程以生成请求
  2. 每个请求调用 /api/loadtest 错误概率为100%
  3. 负载测试端点会抛出一个随机的异常类型
  4. 每个请求需要~1.5秒的延迟可见性

例外类型

模拟每次请求随机抛出以下一种例外类型：

例外类型 描述
NullReferenceException 对象引用未设置为实例
TimeoutException 操作超时
InvalidOperationException 当前状态下不适用的操作
ArgumentException 值超出预期范围
HttpRequestException 发送请求 HTTP 错误
DivideByZeroException 尝试除以零
KeyNotFoundException 词典中没有密钥
IndexOutOfRangeException 指数数组边界外

仪表盘事件日志

每个失败请求会在事件日志中显示：

  • 棕色 - 独特造型以提高可见度
  • ❌ 图标 - 请求失败指示器
  • 例外类型 - 被抛出的特定例外情况
  • 延迟 - 请求时长（毫秒）

参见 Azure 诊断指南 详细 AppLens 分析说明。

⚙️ 可选配置

你可以用 Azure App Service 环境变量自定义应用行为。

健康探针率

控制服务器发送健康探针的频率以测量请求延迟。所有探针都经过 Azure 前端，以捕捉真实的端到端延迟。

环境变量

变量名称 描述 默认
HEALTH_PROBE_RATE 探测间隔以毫秒计。至少100毫秒。 200

通过 Azure CLI设置

# 如果分析器显示重叠请求 CLR 请减慢探测速度az webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \ --设置 HEALTH_PROBE_RATE=400

闲置暂停

当应用程序处于空闲状态（无仪表盘连接或负载测试请求）时，健康探针会自动暂停，以减少不必要的网络流量 Azure前端和 Application Insights 遥测数据。

环境变量

变量名称 描述 默认
IDLE_TIMEOUT_MINUTES 暂停健康探针前的几分钟不活动。当仪表盘打开或收到任何请求时，活动会自动恢复。 20

通过 Azure CLI设置

# 将闲置暂停延长到30分钟az webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \ --设置 IDLE_TIMEOUT_MINUTES=30

自定义页面页脚

设置一个自定义页脚消息，显示在所有页面上。页脚支持 HTML，允许你包含链接。

环境变量

变量名称 描述
PAGE_FOOTER 页脚第二行的HTML内容。如果没有设置，只显示应用描述和构建信息。

示例价值

由 <a href=“https://yoursite.com”>Your Team</a> 创建，用于培训目的

通过 Azure CLI设置

# 通过 CLI 设置页脚Azureaz webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \ --settings'PAGE_FOOTER=由<a href=“https://yoursite.com”>Your Team</a>创建，用于训练'

通过 Azure 门户设定

  1. 在Azure门户中导航到你的App Service
  2. 前往 背景设定 → 环境变量
  3. 咔哒 + 加法
  4. 套装名称： PAGE_FOOTER
  5. 设置值：你的HTML页脚内容
  6. 咔哒 申请，则 确认

GitHub 仓库链接

在侧边栏菜单中显示指向 GitHub 仓库的链接。该链接默认是隐藏的，只有在两个环境变量都配置好后才会出现。

变量名称 描述

🌐 语言选择（界面翻译）

仪表盘界面可以用以下支持的任何语言显示 Azure 认知服务翻译器.翻译在启动时由AI自动生成，并缓存以供后续请求使用。

⚠️ AI翻译通知： 翻译由 Azure 认知服务翻译器（机器翻译）生成，可能不完全准确。技术术语（例如， CPU、 SignalR、 ThreadPool）在英语中被有意保留。

工作原理

  1. 设置 UI_LANGUAGE 环境变量 ISO 639-1 语言代码 （例如， es 西班牙语， fr 对于法语， ja 用于日语）
  2. 启动时，应用会检查该语言是否存在翻译过的本地文件
  3. 如果没有（或者英文源字符串发生了变化），它会调用 Azure Translator 生成一个
  4. 翻译后的文件被缓存——后续启动时使用缓存时，转换成本为零
  5. 仪表盘加载翻译后的字符串并应用到界面上

环境变量

变量名称 描述 默认
UI_LANGUAGE 仪表盘界面的 ISO 639-1 语言代码。设置为 en 英文版（无需翻译）。 en
TRANSLATOR_API_KEY Azure 认知服务翻译器订阅密钥。非英语语言必修。 （未设定）
TRANSLATOR_ENDPOINT 翻译器 API 端点URL。 https://api.cognitive.microsofttranslator.com
TRANSLATOR_REGION Azure 翻译资源的区域。 eastus

  1. 在Azure门户中导航到你的App Service
  2. 前往 背景设定 → 环境变量
  3. 咔哒 + 加法 并集合 名称： UI_LANGUAGE， 值：你的语言代码（例如， es)
  4. 咔哒 + 加法 并集合 名称： TRANSLATOR_API_KEY， 值：你的 Azure 翻译键
  5. 咔哒 + 加法 并集合 名称： TRANSLATOR_REGION， 值：你的翻译资源区域
  6. 咔哒 申请，则 确认 （需要重启）
