☁️ Azure 诊断指南

了解如何使用 Azure App Service 诊断工具诊断性能问题

📊 概述

本指南将解释如何使用 Azure 诊断工具来诊断性能问题模拟器产生的性能问题。

模拟器会产生四种类型的问题：

  1. 高 CPU - 并行自旋环路消耗所有磁芯
  2. 记忆压力 - 大对象堆上的固定分配
  3. 线池饥饿 - 同步通过异步的分断模式
  4. 慢速请求 - 长期请求 CLR 分析器分析

每个问题都可以通过特定的 Azure 工具和技术进行诊断。

🔗 仿真ID相关性

每个仿真实例都被赋予唯一 仿真ID （GUID）实现仪表盘界面与 Azure 诊断之间的端到端关联。

Azure： 相同的仿真ID也出现在服务器端日志中，并可在以下格式中找到：

  • 日志分析 - 使用ID作为过滤器的查询日志
  • 容器/App Service 日志- 按 标准/标准GUID 过滤

故障排除工作流程

  1. 从仪表盘开始模拟
  2. 将鼠标悬停在事件日志中的开始消息上，复制模拟ID
  3. 打开 Azure 门户→ Application Insights →日志（或日志分析）
  4. 使用仿真ID查询日志，查看所有相关的服务器端活动
  5. 将仪表盘事件与服务器指标的时间戳关联起来

提示： 当多个并发模拟运行或仪表盘被多个用户使用时，模拟ID有助于隔离遥测数据。

🔥 诊断高发性 CPU

症状

  • 应用程序变得无响应
  • 请求超时次数增加
  • CPU 指标显示持续高使用率（>80%）

Azure 可用的工具

1. App Service 诊断和解决问题

  1. 在Azure门户中导航到你的App Service
  2. 咔哒 诊断和解决问题 在左侧菜单中
  3. 精选 可用性与性能 > 高 CPU 分析
  4. 回顾时间线和流程分解

2. Application Insights 实时指标

  1. 为你的应用打开Application Insights
  2. 前往 实时指标
  3. 实时观察 CPU 图
  4. 注意与输入请求的相关性

3. CPU 画像（高级）

  1. App Service，前往诊断和解决问题
  2. 搜索“CPU 画像”
  3. 咔哒 收集分析器追踪
  4. 触发 CPU 模拟
  5. 等待收款完成
  6. 下载并分析 .diagsession 文件

注意点

在分析器追踪中，你会看到：- 运行 while 循环的专用线程 - CpuStressService 中的高自有时间 - 线程名称如 CpuStress-*

📊 诊断记忆压力

症状

  • 内存使用随时间增加
  • GC 收集频率增加
  • 由于 GC 暂停，应用速度可能会变慢
  • 最终，OutOfMemoryException

Azure 可用的工具

1. App Service 指标

  1. 导航到你的 App Service
  2. 前往 指标
  3. 添加指标： 记忆工作集, 私有字节
  4. 将时间粒度设置为1分钟
  5. 观察模拟过程中的记忆增长

2. 内存转储分析

  1. 前往 诊断和解决问题
  2. 搜索“内存倾倒”
  3. 咔哒 收集内存转储
  4. 选择“完整数据”以获得详细分析
  5. 下载后在Visual Studio或WinDbg中打开

3. Application Insights

  1. 开放 Application Insights
  2. 前往 性能 > 依赖关系
  3. 寻找与 GC相关的慢节奏模式

注意点

在内存转储中，寻找：- 大对象堆（LOH）中的大字节[]数组 - GCHandle.Alloc 固定对象 - 因钉顶而未被收集的物品

这是MemoryPressureService中有意设计的“坏”代码： VAR 数据 = 新 字节[大小 Megabytes * 1024 * 1024];VAR pinnedHandle = GCHandle.Alloc（data， GCHandleType.Pinned）;固定配置： // 1.自 85KB >起直接进入大型对象堆（LOH） // 2.无法被 GC移动，导致碎片化 // 3.在明确发布前，仍会被记忆保存

🧵 诊断线程池饥饿

症状

  • 请求排队并超时
  • 即使是简单的请求也会延迟飙升
  • 日志中的线程池耗尽警告
  • 健康检查可能失败

Azure 可用的工具

1. Application Insights 依赖关系

  1. 开放 Application Insights
  2. 前往 应用地图
  3. 注意外拨电话延迟较高的情况
  4. 注意“等待线程”的模式

2. App Service 诊断

  1. 前往 诊断和解决问题
  2. 搜索“Threading”
  3. 复习线程数和争用指标

3. 绩效计数器

请监控这些计数器：

  • .NET CLR Thread Pool / # of Threads
  • .NET CLR Thread Pool / Queue Length
  • Process / Thread Count

注意点

线池缺失的迹象：- 线数稳步增加 - 队列长度增长 - 请求时长 - 简单操作的超时

🐌 用 CLR Profiler 诊断慢速请求

慢速请求模拟器使用专门设计以便于CLR分析器跟踪中易于识别的sync-over-async模式生成长时间请求（每个请求约25秒）。

目的

该模拟设计用于 培训开发人员识别阻塞电话 在画像师的痕迹中。与线程池饥饿模拟（导致许多短块）不同，这会创建单个长请求，这些请求在调用栈中清晰可见。

Azure 可用的工具

1. Azure 自动治疗，触发缓慢请求

  1. 在Azure门户中导航到你的App Service
  2. 前往 诊断和解决问题
  3. 搜索“自动治疗”
  4. 配置慢速请求触发（例如，请求>20秒）

2. CLR 画像师 / Azure 画像师

  1. 先触发慢请求模拟——这会使线程池减少，减慢健康探针速率并减少走线中的噪声
  2. App Service，前往诊断和解决问题
  3. 搜索“Profiler”
  4. 开始一个 CPU 画像师 追踪（建议60-120秒）
  5. 下载并分析该追踪

CLR分析师应关注什么

该服务使用三种不同的 sync-over-async 模式，分别为 有意描述的方法名称:

情景一：简单阻挡

寻找方法： FetchDataSync_BLOCKING_HERE, ProcessDataSync_BLOCKING_HERE, SaveDataSync_BLOCKING_HERE

在分析器中，你会看到花费时间： → SlowRequestService.FetchDataSync_BLOCKING_HERE →线索。睡眠

场景二：嵌套同步方法

寻找以下结尾的方法： _BLOCKS_INTERNALLY

这些是通过 Thread.Sleep 内部阻断的同步方法： → ValidateOrderSync_BLOCKS_INTERNALLY → CheckInventorySync_BLOCKS_INTERNALLY → ProcessPaymentSync_BLOCKS_INTERNALLY → SendConfirmationSync_BLOCKS_INTERNALLY

情景三：数据库/HTTP 模式

寻找以下结尾的方法： Sync_SYNC_BLOCK

模拟数据库和 HTTP 阻断呼叫： → GetCustomerFromDatabaseSync_SYNC_BLOCK → GetOrderHistoryFromDatabaseSync_SYNC_BLOCK → CheckInventoryServiceSync_SYNC_BLOCK → GetRecommendationsFromMLServiceSync_SYNC_BLOCK → BuildResponseSync_SYNC_BLOCK

💡 关键洞察：排队时间与执行时间

如果你在请求开始前看到“无事件发出”的较大空白：

  • 排队时间： 线程池处于缺电状态，所以你的请求会在队列中等待线程（例如，20+秒）。
  • 执行时间： 一旦分配了线程，方法才会最终运行。

示例： 用户等待47秒，但日志显示只有25秒。缺失的22元卡都在排队里花光了！

💡 ETW心跳

模拟器日志显示“产生迹迹噪声......”每隔500毫秒清除ETW缓冲区，确保长请求不会在跟踪中丢失。

💥 诊断应用程序崩溃

症状

  • 应用突然变得不可用
  • HTTP 502/503 错误返回给客户端
  • Azure 自动重启应用程序
  • 事件日志显示进程终止

Azure 碰撞监测

Azure App Service 内置了崩溃监控功能，可以在应用崩溃时自动捕捉内存转储。

启用碰撞监控

  1. 在Azure门户中导航到你的App Service
  2. 前往 诊断和解决问题
  3. 搜索“Crash Monitoring”
  4. 启用崩溃转储收集

⚠️ 重要提示：存储账户限制

在某些 Azure 环境中，由于存储账户的安全限制，崩溃监控可能无法正常工作。如果你不能使用崩溃监控，也可以手动收集内存转储，使用 程序转储 来自Kudu控制台。

使用 ProcDump 进行手动转储收集

程序转储 是 Sysinternals 的一个命令行工具，可以监控应用程序并生成崩溃转储。它可以通过Kudu控制台在 Azure App Service 上购买。

访问Kudu控制台

  1. 导航到你的 App Service
  2. 前往 开发工具 > 高级工具
  3. 咔哒 走 开启库杜
  4. 精选 调试控制台 > CMD 或 PowerShell

常见的 ProcDump 命令

# 按名称写入进程的完整内存转储procdump -马 w3wp.exe# 当发生未处理异常时，写入完整转储procdump -马 -e w3wp.exe# 在第一次或第二次机会例外时写一个完整的倾倒procdump -马 -e 1 w3wp.exe# 最多写10个dump，每个例外一个procdump -马 -n 10 -e 1 w3wp.exe# 当 CPU 超过80%持续10秒时写一个dumpprocdump -马 -c 80 -s 10 w3wp.exe# 当内存超过1GB时写入转储procdump -马 -M 1024 w3wp.exe# 当检测到悬挂窗口时写一则废料procdump -马 -h w3wp.exe

ProcDump 选项参考

选项 描述
-ma 全内存转储（Full dump）
-mm 迷你倾倒（默认，体积较小）
-mp MiniPlus的转储（细节丰富，但比全尺寸小10-75%）
-e 在未处理异常时转储（add 1 第一次机会）
-c CPU 阈值百分比
-m 内存提交阈值（MB）
-n 退出前需要写入的转储数量
-s 倾倒前连续秒数（默认10秒）
-h 挂窗上的倾倒
-t 进程终止时的倾倒

下载转储文件

  1. 在Kudu中，导航到导出文件创建的文件夹
  2. 点击旁边的下载图标 .dmp 文件
  3. 在Visual Studio、WinDbg或其他调试器中打开转储文件

❌ 诊断失败请求（HTTP 5xx）

HTTP 500错误是影响应用可用性的最常见问题之一。该模拟生成随机异常，用于练习在 Azure中诊断失败请求。

要模拟什么

  1. 在仪表盘中，点击 ❌ 启动失败请求
  2. 等待请求完成（每个请求需时~1.5秒）
  3. 在事件日志中观察异常类型（棕色条目）

诊断 AppLens

  1. 在 Azure 门户，前往你的 App Service
  2. 导航至此 诊断和解决问题
  3. 搜索 “应用错误” 或 “HTTP 500”
  4. AppLens 将展示：
    • 误差率随时间变化
    • 最常见的例外类型
    • 每次故障的栈道跟踪
    • 与其他症状的相关性

诊断 Application Insights

  1. 在 Azure 门户中，访问你的 Application Insights 资源
  2. 咔哒 故障 在左侧菜单中
  3. 请审查故障时间线和例外情况
  4. 点击个别例外查看：
    • 全异常栈跟踪
    • 请求详情（URL、时长、响应代码）
    • 自定义属性与遥测

常见的例外类型

例外 现实事业 如何修复
NullReferenceException 访问空对象的性质 添加空检查，使用可空引用类型
TimeoutException 外部依赖太慢了 增加超时，添加断路器
InvalidOperationException 处于错误状态的物体无法操作 检查前提条件，验证状态
HttpRequestException 外部 API 呼叫失败 添加重试逻辑，检查连接性
KeyNotFoundException 缺少配置或缓存密钥 使用TryGetValue，先检查存在性

💡 专业提示：创建错误率提醒

设置一个 Azure 监控提醒，当 HTTP 5xx错误率超过阈值时通知你：

  1. 去你的 App Service → 警报 → 创建警报规则
  2. 信号： HTTP 服务器错误
  3. 阈值：动态或静态（例如每分钟>5个）
  4. 操作：发送邮件、短信或网页挂钩通知
