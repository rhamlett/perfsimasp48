🚀 Azure Guia de Implantação

Implante o Performance Problem Simulator para Azure App Service usando GitHub Ações com autenticação OIDC

Visão geral

Este guia guia como implantar o Performance Problem Simulator para Azure App Service usando GitHub Ações com autenticação OpenID Connect (OIDC). OIDC elimina a necessidade de armazenar credenciais como segredos.

Pré-requisitos

  • Azure assinatura com permissões para criar recursos
  • Azure permissões AD/Entra ID para criar Registros de Aplicativos
  • GitHub conta
  • Azure CLI instalado (opcional, para o método CLI)

O que você vai criar

RecursoPropósito
Azure App ServiceHospeda a aplicação Performance Problem Simulator
Azure Registro do AD AppIdentidade para GitHub Ações OIDC
Credencial FederadaLinks GitHub repositório para Azure AD
Atribuição de FunçõesConcede permissões de implantação

1 Crie Azure App Service

Opção A: Azure Portal

  1. Navegar para Serviços de Aplicativos
    • Vá para Azure Portal
    • Procure por "App Services" e selecione essa opção
    • Click + Criar
  2. Configure o Básico
    AmbientaçãoValor
    AssinaturaSua assinatura
    Grupo de RecursosCriar novos ou usar os existentes
    Nomeyour-app-name (deve ser globalmente único)
    PublicarCódigo
    Pilha de tempo de execuçãoSelecione o tempo de execução da sua aplicação
    Sistema OperacionalWindows ou Linux (conforme apropriado para o seu tempo de execução)
    RegiãoSua região preferida
  3. Configure App Service Plano
    AmbientaçãoRecomendação
    SKUB1 Básico ou superior

    ⚠️ Importante: A camada Livre (F1) não suporta WebSockets, que são necessários para atualizações em tempo real de SignalR dashboards.

  4. Revise e Crie - Clique para criar o recurso

Opção B: Azure CLI

# Login to Azure
az login

# Set variables
RESOURCE_GROUP="your-resource-group"
APP_NAME="your-app-name"
LOCATION="eastus"
APP_SERVICE_PLAN="your-app-plan"
RUNTIME="your-runtime"  # e.g., "dotnet:8", "node:20-lts", "python:3.11"

# Create resource group
az group create --name $RESOURCE_GROUP --location $LOCATION

# Create App Service Plan (B1 for WebSocket support)
az appservice plan create \
  --name $APP_SERVICE_PLAN \
  --resource-group $RESOURCE_GROUP \
  --location $LOCATION \
  --sku B1

# Create Web App with your runtime
az webapp create \
  --name $APP_NAME \
  --resource-group $RESOURCE_GROUP \
  --plan $APP_SERVICE_PLAN \
  --runtime $RUNTIME

Habilitar WebSockets

Após a criação, ative o WebSockets para comunicação em tempo real SignalR :

  1. Vá para sua App Service → Configuração → Configurações gerais
  2. Cenário Soquetes web para Em
  3. Click Salvar

Ou via CLI:

az webapp config set \
  --name $APP_NAME \
  --resource-group $RESOURCE_GROUP \
  --web-sockets-enabled true

2 Criar Azure Registro de Aplicativo AD

GitHub Actions usa OpenID Connect (OIDC) para autenticar com Azure sem armazenar credenciais.

Criar o Registro do App

  1. Navegue até Inscrições de Aplicativos
    • Vá para Azure Portal
    • Procure por "Registros de aplicativos" e selecione
    • Click + Novo registro
  2. Configurar o registro
    AmbientaçãoValor
    Nomegithub-your-app-deploy (nome descritivo para sua identidade de implantação)
    Tipos de contas suportadosContas neste diretório organizacional apenas
    URI de redirecionamentoDeixe vazio
  3. Clique em Registrar

Registre as identidades

Do Visão geral Page, copie esses valores (você vai precisar deles para GitHub segredos):

ValorGitHub Nome Secreto
ID de aplicação (cliente)AZURE_CLIENT_ID
ID de diretório (locatário)AZURE_TENANT_ID

Obtenha o ID da Assinatura

  1. Vá para Assinaturas no Azure Portal
  2. Selecione sua assinatura
  3. Cópia do ID de assinatura → É isso AZURE_SUBSCRIPTION_ID

3 Configurar Credenciais Federadas

Credenciais federadas permitem que GitHub Ações se autentiquem sem armazenar segredos.

  1. Navegue até o Registro do seu App
    • Vá para Certificados e segredos
    • Selecionar Credenciais federadas Tab
    • Click + Adicionar credencial
  2. Configurar a Credencial
    AmbientaçãoValor
    Cenário de credencial federadaGitHub Ações que mobilizam recursos Azure
    OrganizaçãoSeu nome de usuário GitHub ou organização
    RepositórioNome do seu repositório
    Tipo de entidadeRamo
    GitHub nome da filialmain
    Nomegithub-main-branch
  3. Clique em Adicionar

💡 Nota: Se você também quiser implantar a partir de pull requests ou outros branchs, adicione credenciais federadas adicionais com o tipo de entidade apropriado.

4 Conceda Azure Permissões

O Registro de Aplicativos precisa de permissão para ser implantado no seu App Service.

Atribuir Papel de Contribuinte

  1. Navegue até o seu App Service
    • Vá para Controle de acesso (IAM)
    • Click + Adicionar → Adicionar atribuição de funções
  2. Configurar Atribuição de Funções
    AmbientaçãoValor
    FunçãoColaborador
    Atribuir acesso aUsuário, grupo ou principal de serviço
    MembrosProcure pelo nome do seu Registro de App
  3. Clique em Revisar + atribuir

5 Configurar GitHub Segredos

Clone o Repositório

  1. Clone o Repositório
    • Vá para o repositório Performance Problem Simulator da sua pilha
    • Clone o repositório para sua máquina local

Adicionar GitHub Segredos

  1. Navegar até Configurações do Repositório
    • Acesse seu repositório no GitHub
    • Click Configurações → Segredos e variáveis → Ações
    • Click Novo segredo de repositório
  2. Adicione os Seguintes Segredos
    Nome SecretoValorDescrição
    AZURE_CLIENT_IDxxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxxID do Cliente de Registro do App
    AZURE_TENANT_IDxxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxxAzure Diretório AD/ID de Tenant
    AZURE_SUBSCRIPTION_IDxxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxxID de assinatura Azure

💡 Sem Segredo do Cliente:OIDC credenciais federadas eliminam a necessidade de armazenar um segredo do cliente. Os três IDs acima são suficientes.

Atualize o Fluxo de Trabalho (se necessário)

Atualização .github/workflows/deploy.yml com seu nome App Service e versão em tempo de execução:

env:
  AZURE_WEBAPP_NAME: your-app-service-name  # Your App Service name
  # Add any runtime-specific version variables as needed

6 Implantar

Implantação Automática

O deployment é acionado automaticamente quando você empurra para o main Ramo:

git add .
git commit -m "Your changes"
git push origin main

Implantação Manual

  1. Acesse seu repositório no GitHub
  2. Click Ações Tab
  3. Selecionar Desdobrar para Azure App Service
  4. Click Executar o fluxo de trabalho → Executar o fluxo de trabalho

Verificar a implantação

  1. Abra a URL do aplicativo: https://<your-app-name>.azurewebsites.net
  2. Verificar o Painel: Métricas em tempo real devem ser atualizadas via SignalR, o status deve mostrar "Conectado"
  3. Endpoint de Saúde do Teste: 
    curl https://<your-app-name>.azurewebsites.net/api/health

Configuração Opcional

Você pode personalizar o comportamento da aplicação usando Azure App Service variáveis de ambiente.

Taxa de Sondagem de Saúde

Controle com que frequência o servidor envia sondas de saúde para medir a latência das requisições. Todas as sondas são roteadas pelo frontend Azure para capturar latência realista de ponta a ponta.

Variável Ambiente

Nome da variável Descrição Padrão
HEALTH_PROBE_RATE Intervalo de sonda em milissegundos. Mínimo 100ms. 200

Configuração via Azure CLI

# Slow down probes if CLR profiler shows overlapping requests
az webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \
  --settings HEALTH_PROBE_RATE=400

Tempo de inatividade

Quando a aplicação está ociosa (sem conexões no painel ou requisições de teste de carga), as sondas de saúde são automaticamente suspensas para reduzir o tráfego desnecessário de rede para a interface e a telemetria Application Insights da Azure.

Variável Ambiente

Nome da variável Descrição Padrão
IDLE_TIMEOUT_MINUTES Minutos de inatividade antes de suspender as sondas de saúde. A atividade é retomada automaticamente quando o painel é aberto ou qualquer solicitação é recebida. 20

Configuração via Azure CLI

# Extend idle timeout to 30 minutes
az webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \
  --settings IDLE_TIMEOUT_MINUTES=30

Rodapé de página personalizado

Defina uma mensagem personalizada no rodapé que apareça em todas as páginas. O rodapé suporta HTML, permitindo incluir links.

Variável Ambiente

Nome da variável Descrição
PAGE_FOOTER Conteúdo HTML para a segunda linha do rodapé. Se não estiver definido, apenas a descrição do app e as informações da build são mostradas.

Exemplo de Valor

Created by <a href="https://yoursite.com">Your Team</a> for training purposes

Configuração via Azure CLI

# Set the footer via Azure CLI
az webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \
  --settings 'PAGE_FOOTER=Created by <a href="https://yoursite.com">Your Team</a> for training purposes'

Ambientação via Azure Portal

  1. Navegue até sua App Service no Portal Azure
  2. Vá para Configurações → Variáveis do ambiente
  3. Click + Adicionar
  4. Nome do Conjunto: PAGE_FOOTER
  5. Valor de Definição: Seu conteúdo do rodapé HTML
  6. Click Candidate-se, então Confirme

GitHub Link do Repositório

Exiba um link para o repositório GitHub no menu lateral. O link é oculto por padrão e só aparece quando ambas as variáveis de ambiente estão configuradas.

Variáveis Ambientais

Nome da variável Descrição
GITHUB_USER_NAME GitHub nome de usuário ou organização (por exemplo, contoso)
GITHUB_REPO_NAME Nome do repositório (por exemplo, perf-simulator)

Configuração via Azure CLI

# Enable GitHub link in the sidebar menu
az webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \
  --settings GITHUB_USER_NAME=contoso GITHUB_REPO_NAME=perf-simulator

Solução de problemas

SignalR Falha na Conexão

Sintomas: O painel mostra "Desconectado", os gráficos não atualizam em tempo real

Soluções:

  • Garanta que os WebSockets estejam ativados em App Service Configuração
  • Upgrade do nível Gratuito para Básico ou superior
  • Verifique se o ARR Affinity está ativado (Configurações → Configuração → Configurações Gerais)

OIDC Falha na Autenticação

Sintomas:GitHub Ações falham na etapa "Logar para Azure"

Confere:

  • Verifique se os três segredos estão corretamente definidos
  • Confirme que o assunto da credencial federada corresponde ao seu repositório/ramo: repo:USERNAME/REPO:ref:refs/heads/main
  • Garanta que o Registro de Aplicativos tenha o papel de Contribuidor na App Service

O aplicativo trava na inicialização

Sintomas: O aplicativo mostra "Erro de Aplicação" ou não responde

Confere:

  • Ver fluxo de registro no portal Azure (App Service → fluxo de registro)
  • Ativear o Registro de Aplicações em App Service → App Service Logs
  • Verifique se a versão em tempo de execução corresponde aos requisitos da sua aplicação
  • Verifique variáveis de configuração ou ambiente ausentes

Ver Registros

# Stream live logs
az webapp log tail --name $APP_NAME --resource-group $RESOURCE_GROUP

# View deployment logs
az webapp log deployment show --name $APP_NAME --resource-group $RESOURCE_GROUP

Referência Rápida - Frase de CLI

Script de configuração completo (personalize variáveis no topo):

# Full setup (customize variables at top)
RESOURCE_GROUP="your-resource-group"
APP_NAME="your-app-name"
LOCATION="eastus"
RUNTIME="your-runtime"  # e.g., "dotnet:8", "node:20-lts", "python:3.11"
GITHUB_REPO="YOUR_USERNAME/YOUR_REPO"

# Create resources
az group create -n $RESOURCE_GROUP -l $LOCATION
az appservice plan create -n "${APP_NAME}-plan" -g $RESOURCE_GROUP -l $LOCATION --sku B1
az webapp create -n $APP_NAME -g $RESOURCE_GROUP -p "${APP_NAME}-plan" --runtime $RUNTIME
az webapp config set -n $APP_NAME -g $RESOURCE_GROUP --web-sockets-enabled true

# Create App Registration and get IDs
APP_ID=$(az ad app create --display-name "github-${APP_NAME}-deploy" --query appId -o tsv)
az ad sp create --id $APP_ID
TENANT_ID=$(az account show --query tenantId -o tsv)
SUB_ID=$(az account show --query id -o tsv)

# Create federated credential
az ad app federated-credential create --id $APP_ID --parameters "{
  \"name\": \"github-main-branch\",
  \"issuer\": \"https://token.actions.githubusercontent.com\",
  \"subject\": \"repo:${GITHUB_REPO}:ref:refs/heads/main\",
  \"audiences\": [\"api://AzureADTokenExchange\"]
}"

# Assign permissions
APP_SERVICE_ID=$(az webapp show -n $APP_NAME -g $RESOURCE_GROUP --query id -o tsv)
az role assignment create --assignee $APP_ID --role Contributor --scope $APP_SERVICE_ID

# Output secrets for GitHub
echo "AZURE_CLIENT_ID=$APP_ID"
echo "AZURE_TENANT_ID=$TENANT_ID"
echo "AZURE_SUBSCRIPTION_ID=$SUB_ID"
