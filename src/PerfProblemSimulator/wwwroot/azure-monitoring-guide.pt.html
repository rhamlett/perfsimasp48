☁️ Azure Guia de Diagnóstico

Aprenda a diagnosticar problemas de desempenho usando ferramentas de diagnóstico Azure App Service

📊 Visão geral

Este guia explica como usar Azure ferramentas de diagnóstico para diagnosticar os problemas de desempenho criados pelo Performance Problem Simulator.

O simulador cria quatro tipos de problemas:

  1. Alto CPU - Laços de spin paralelos que consomem todos os núcleos
  2. Pressão de Memória - Alocações fixadas no Grande Monte de Objetos
  3. Falta de Poças de Fios - Padrões de bloqueio sincronizado sobre assíncrono
  4. Pedidos Lentos - Solicitações de longa duração para análise de CLR Profiler

Cada problema pode ser diagnosticado usando ferramentas e técnicas específicas Azure .

🔗 Correlação do ID da Simulação

Cada instância de simulação recebe um único ID de simulação (GUID) que permite correlação de ponta a ponta entre a interface do painel e Azure diagnósticos.

Encontrando IDs de Simulação

No Painel: Passe o mouse sobre qualquer mensagem relacionada à simulação no Registro de Eventos (marcada com um sublinhado pontilhado) para ver o ID completo da Simulação em uma dica de ferramenta.

Em Azure: O mesmo ID de simulação é emitido nos logs do lado do servidor e pode ser encontrado em:

  • Application Insights - Rastreamentos de busca e eventos personalizados por ID de Simulação
  • Análise de Logs - Logs de consulta usando o ID como filtro
  • Troncos de Contêineres/App Service - Filtre stdout/stderr pelo GUID

Fluxo de Trabalho de Solução de Problemas

  1. Inicie uma simulação a partir do painel
  2. Passe o mouse sobre a mensagem de início no Log de Eventos para copiar o ID da Simulação
  3. Abrir Azure Portal → Application Insights → Logs (ou Análise de Logs)
  4. Consulte logs usando o ID de simulação para ver toda a atividade relacionada do lado do servidor
  5. Correlacione carimbos de tempo entre eventos do painel e métricas de servidor

Dica: O ID de Simulação ajuda a isolar a telemetria ao executar múltiplas simulações simultâneas ou quando o painel é usado por múltiplos usuários.

🔥 Diagnosticando Alta CPU

Sintomas

  • A aplicação deixa de responder
  • Aumento dos tempos de espera dos pedidos
  • CPU métrica mostra uso sustentado e alto (>80%)

Azure Ferramentas a Usar

1. App Service Diagnosticar e Resolver Problemas

  1. Navegue até sua App Service no Portal Azure
  2. Click Diagnosticar e resolver problemas No menu esquerdo
  3. Selecionar Disponibilidade e desempenho > Análise de Alta CPU
  4. Revise o cronograma e a divisão do processo

2. Application Insights Métricas ao Vivo

  1. Abra Application Insights para seu aplicativo
  2. Vá para Métricas ao Vivo
  3. Observe o gráfico CPU em tempo real
  4. Note a correlação com as solicitações recebidas

3. CPU Perfilamento (Avançado)

  1. Em App Service, vá para Diagnosticar e resolver problemas
  2. Procure por "CPU Perfilando"
  3. Click Colete Rastreamento do Perfilador
  4. Acione a simulação CPU
  5. Espere a coleta concluir
  6. Baixe e analise o .diagsession arquivo

O que procurar

No rastreamento do perfilador, você verá:- Threads dedicadas executando um loop while - Alto tempo de auto-autonomia no CpuStressService - Nomes de threads como CpuStress-*

Padrão de Código (O Que Está Errado)

O serviço utiliza threads dedicados para evitar privar o Thread Pool, garantindo que o painel permaneça responsivo mesmo sob 100% de carga. 📊 Diagnóstico da Pressão da Memória

Sintomas

  • Aumento do uso de memória ao longo do tempo
  • GC frequência de coleta aumenta
  • A aplicação pode ficar mais lenta devido a pausas GC
  • Eventualmente, OutOfMemoryException

Azure Ferramentas a Usar

1. App Service Métricas

  1. Navegue até o seu App Service
  2. Vá para Métricas
  3. Adicionar métricas: Conjunto de trabalho de memória, Bytes Privados
  4. Defina a granularidade do tempo para 1 minuto
  5. Observe o crescimento da memória durante a simulação

2. Análise de Despejo de Memória

  1. Vá para Diagnosticar e resolver problemas
  2. Procure por "Memory Dump"
  3. Click Coletar Memória Dump
  4. Escolha "Despejo completo" para uma análise detalhada
  5. Baixar e abrir no Visual Studio ou WinDbg

3. Application Insights

  1. Aberto Application Insights
  2. Vá para Desempenho > Dependências
  3. Procure padrões lentos relacionados à GC

O que procurar

Em um despejo de memória, procure por:- Arrays de bytes grandes[] no Grande Monte de Objetos (LOH) - GCHandle.Alloc objetos fixados - Objetos que não estão sendo coletados devido ao fixamento

Padrão de Código (O Que Está Errado)

Este é o código "ruim" intencional no MemoryPressureService: Var dados = Novo byte[tamanhoMegabytes * 1024 * 1024];Var pinnedHandle = GCHandle.Alloc(data, GCHandleType.Pinned);A alocação fixada: // 1. Vai direto para o Large Object Heap (LOH) desde > 85KB // 2. Não pode ser movido por GC, causando fragmentação // 3. Permanece na memória até ser explicitamente lançado

🧵 Diagnosticando a Falta de Pool de Threads

Sintomas

  • Solicitações entram em fila e tempo limite
  • Picos de latência até mesmo para requisições simples
  • Avisos de exaustão de pool de rosca nos registros
  • Exames de saúde podem falhar

Azure Ferramentas a Usar

1. Application Insights Dependências

  1. Aberto Application Insights
  2. Vá para Mapa de Aplicação
  3. Procure alta latência nas chamadas de saída
  4. Note o padrão "aguardando fio"

2. App Service Diagnósticos

  1. Vá para Diagnosticar e resolver problemas
  2. Procure por "Threading"
  3. Revise a contagem de tópicos e métricas de contenda

3. Contadores de Desempenho

Monitore estes contadores:

  • .NET CLR Thread Pool / # of Threads
  • .NET CLR Thread Pool / Queue Length
  • Process / Thread Count

O que procurar

Sinais de falta de poças de fios:- Número de threads aumentando de forma constante - Crescimento do comprimento da fila - Longas durações de solicitação - Tempos de espera em operações simples

Padrão de Código (O Que Está Errado)

Este é o código "ruim" intencional no ThreadBlockService:Tarefa.Correr(assíncrono () => { RUIM: Bloqueando uma thread pool de threads com espera síncrona! Este é o "sync-over-async" anti-padrão.Tarefa.Delay(delayMs, cancellationToken). Espere(); Problema: Cada chamada para . Wait() bloqueia uma thread pool de threads até que o atraso termine. Com pedidos suficientes em simultâneo, O pool de threads se esgota e novos pedidos precisam esperar.});

🐌 Diagnosticando Solicitações Lentas com CLR Profiler

O Slow Request Simulator gera requisições de longa duração (~25 segundos cada) usando padrões sync-over-async especificamente projetados para serem facilmente identificáveis em CLR rastreamentos do Profiler.

Propósito

Esta simulação foi projetada para Treinando desenvolvedores para identificar chamadas de bloqueio em rastreamentos de perfiladores. Diferente da simulação de Starvation do Thread Pool (que causa muitos blocos curtos), essa cria requisições longas individuais que são claramente visíveis em pilhas de chamadas.

Azure Ferramentas a Usar

1. Azure Cura Automática com Gatilho de Pedido Lento

  1. Navegue até sua App Service no Portal Azure
  2. Vá para Diagnosticar e resolver problemas
  3. Procure por "Auto-Cura"
  4. Configure gatilhos de requisição lenta (por exemplo, requisições > 20 segundos)

2. CLR Perfilador / Perfilador Azure

  1. Acione primeiro a simulação de solicitação lenta — isso priva o pool de threads, o que desacelera a taxa de saúde da sonda e reduz o ruído no rastreio
  2. Em App Service, vá para Diagnosticar e resolver problemas
  3. Procure por "Profiler"
  4. Comece um CPU Profiler Trace (recomendado 60-120 segundos)
  5. Baixe e analise o rastreio

O que procurar no CLR Profiler

O serviço utiliza três padrões de sync-over-async diferentes com Nomes de métodos intencionalmente descritivos:

Cenário 1: Bloqueio Simples

Procure por métodos: FetchDataSync_BLOCKING_HERE, ProcessDataSync_BLOCKING_HERE, SaveDataSync_BLOCKING_HERE

No profiler, você verá o tempo gasto em: → SlowRequestService.FetchDataSync_BLOCKING_HERE → Thread. Sleep

Cenário 2: Métodos de Sincronização Aninhados

Procure métodos que terminam em: _BLOCKS_INTERNALLY

Esses são métodos síncronos que bloqueiam internamente usando o Thread.Sleep: → ValidateOrderSync_BLOCKS_INTERNALLY → CheckInventorySync_BLOCKS_INTERNALLY → ProcessPaymentSync_BLOCKS_INTERNALLY → SendConfirmationSync_BLOCKS_INTERNALLY

Cenário 3: Banco de Dados/ PadrãoHTTP

Procure métodos que terminam em: Sync_SYNC_BLOCK

Banco de dados simulado e HTTP bloqueando chamadas: → GetCustomerFromDatabaseSync_SYNC_BLOCK → GetOrderHistoryFromDatabaseSync_SYNC_BLOCK → CheckInventoryServiceSync_SYNC_BLOCK → GetRecommendationsFromMLServiceSync_SYNC_BLOCK → BuildResponseSync_SYNC_BLOCK

💡 Principais Insights: Tempo de Fila vs Tempo de Execução

Se você perceber uma grande lacuna onde "Nenhum evento emitido" aparece antes do início da solicitação:

  • Tempo de Fila: O Pool de Threads está esfomeado, então seu pedido ficou na fila esperando por um thread (por exemplo, 20+ segundos).
  • Tempo de Execução: Uma vez que um fio era atribuído, o método finalmente rodava.

Exemplo: O usuário espera 47s, mas o log mostra apenas 25s de duração. Os 22 anos que faltam foram gastos na fila!

💡 Batimentos cardíacos ETW

O simulador registra "Gerando ruído de traço..." a cada 500ms para esvaziar buffers ETW, garantindo que requisições longas não se percam em rastros.

💥 Diagnóstico de Falhas de Aplicação

Sintomas

  • A inscrição de repente fica indisponível
  • HTTP erros 502/503 retornados aos clientes
  • Azure reinicia automaticamente o aplicativo
  • Registros de eventos mostram a terminação do processo

Azure Monitoramento de Falhas

Azure App Service inclui o Monitoramento de Falhas embutido que pode capturar automaticamente os dumps de memória quando seu aplicativo trava.

Ativando o Monitoramento de Falhas

  1. Navegue até sua App Service no Portal Azure
  2. Vá para Diagnosticar e resolver problemas
  3. Procure por "Monitoramento de Falhas"
  4. Ativar a coleta de despejos de falha

⚠️ Importante: Restrições de Conta de Armazenamento

Em alguns ambientes Azure , o Monitoramento de Falhas pode não funcionar devido a restrições de segurança nas contas de armazenamento. Se você não puder usar o Monitoramento de Falhas, pode coletar dumps de memória manualmente usando ProcDump do console Kudu.

Usando o ProcDump para coleta manual de dumps

ProcDump é uma ferramenta de linha de comando do Sysinternals que pode monitorar aplicações e gerar despejos de falha. Está disponível no Azure App Service pelo console Kudu.

Acessando o Console Kudu

  1. Navegue até o seu App Service
  2. Vá para Ferramentas de Desenvolvimento > Ferramentas Avançadas
  3. Click Vai para abrir Kudu
  4. Selecionar Console de depuração > CMD ou PowerShell

Comandos comuns de ProcDump

# Escreva um despejo completo de memória de um processo pelo nomeprocdump -ma w3wp.exe# Escreva um dump completo quando ocorre uma exceção não tratadaprocdump -ma -e w3wp.exe# Escreva um despejo completo na exceção de 1ª ou 2ª chanceprocdump -ma -e 1 w3wp.exe# Escreva até 10 dumps, um por exceçãoprocdump -ma -n 10 -e 1 w3wp.exe# Escreva um dump quando CPU ultrapassar 80% por 10 segundosprocdump -ma -c 80 -s 10 w3wp.exe# Escreva um dump quando a memória ultrapassar 1GBprocdump -ma -m 1024 w3wp.exe# Escreva um despejo quando uma janela pendurada for detectadaprocdump -ma -h w3wp.exe

Referência das Opções de ProcDump

Opção Descrição
-ma Despejo completo (toda a memória)
-mm Mini dump (padrão, tamanho menor)
-mp Despejo do MiniPlus (detalhado, mas 10-75% menor que o completo)
-e Despejar na exceção não tratada (adicionar 1 para primeira chance)
-c CPU percentual de limiar
-m Limiar de commit de memória em MB
-n Número de dumps a escrever antes de sair
-s Segundos consecutivos antes do dump (padrão 10)
-h Despejar na janela pendurada
-t Despejo na terminação do processo

Baixando o arquivo de despejo

  1. No Kudu, navegue até a pasta onde o dump foi criado
  2. Clique no ícone de download ao lado do .dmp arquivo
  3. Abra o dump no Visual Studio, WinDbg ou outro depurador

❌ Diagnosticando Pedidos Falhados (HTTP 5xx)

Erros HTTP 500 são um dos problemas mais comuns que impactam a disponibilidade dos aplicativos. Essa simulação gera exceções aleatórias para praticar o diagnóstico de solicitações falhadas em Azure.

O que simular

  1. No painel, clique ❌ Iniciar Solicitações Falhadas
  2. Espere a conclusão dos pedidos (cada um leva ~1,5 segundos)
  3. Observar tipos de exceção no Registro de Eventos (entradas marrom)

Diagnosticando com AppLens

  1. No Azure Portal, vá até sua App Service
  2. Navegar para Diagnosticar e resolver problemas
  3. Busca por "Erros de Aplicação" ou "HTTP 500"
  4. AppLens mostrará:
    • Taxa de erro ao longo do tempo
    • Tipos de exceção mais comuns
    • Traços de pilha para cada falha
    • Correlação com outros sintomas

Diagnosticando com Application Insights

  1. No Azure Portal, acesse seu recurso Application Insights
  2. Click Falhas No menu esquerdo
  3. Revise a linha do tempo das falhas e a divisão das exceções
  4. Clique em exceções individuais para ver:
    • Rastreamento de pilha de exceção completa
    • Detalhes da solicitação (URL, duração, código de resposta)
    • Propriedades personalizadas e telemetria

Tipos de exceção comuns que você verá

Exceção Causa do Mundo Real Como Consertar
NullReferenceException Acessando a propriedade do objeto nulo Adicionar verificações de nulo, usar tipos de referência anuláveis
TimeoutException Dependência externa lenta demais Aumentar o timeout, adicionar disjuntor
InvalidOperationException Objeto em estado errado para operação Verifique pré-condições, valide o estado
HttpRequestException Chamada API externa falhou Adicionar lógica de retentativa, verificar a conectividade
KeyNotFoundException Falta da chave de configuração ou cache Use o TryGetValue, verifique a existência primeiro

💡 Dica de Profissional: Criando Alertas de Taxa de Erro

Configure um alerta Azure Monitor para avisar quando a taxa de erro 5xx do HTTP ultrapassar um limite:

  1. Vá para sua App Service → Alertas → Regra de criar alerta
  2. Sinal: Erros do servidor Http
  3. Limiar: Dinâmico ou Estático (por exemplo, > 5 por minuto)
  4. Ação: Notificação por e-mail, SMS ou webhook
