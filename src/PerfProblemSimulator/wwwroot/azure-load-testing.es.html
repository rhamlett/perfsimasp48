📈 Azure Guía de pruebas de carga

Endpoint de pruebas de carga diseñado para Azure integración de pruebas de carga

🔬 Resumen

El Simulador de Problemas de Rendimiento incluye un punto final dedicado para pruebas de carga diseñado para usarse con Azure Pruebas de Carga. Este extremo simula el realista Comportamiento de la aplicación que se degrada de forma elegante bajo carga.

El /api/loadtest Endpoint está diseñado para:

  • Realizar trabajo sostenido de CPU (ciclos SHA256 entrelazados a ~50% CPU por hilo)
  • Mantener los búferes de memoria durante toda la duración de la solicitud (crea presión visible en la memoria)
  • Se degrada de forma elegante a medida que aumentan las solicitudes concurrentes (patrón de límite suave)
  • Lanzar excepciones aleatorias tras el umbral configurable (por defecto: 120 segundos con 20% de probabilidad)
  • Finalmente activó el tiempo muerto de 230 segundos de Azurebajo carga extrema

⚙️ Configuración Azure pruebas de carga

  1. En el Portal Azure, crea un recurso de Pruebas de Carga Azure
  2. Crea una nueva prueba y configura la URL de destino usando una de estas opciones:

GET con parámetros de consulta

Usa la URL de destino directamente — no se requiere un script JMeter:

# With defaults
https://your-app.azurewebsites.net/api/loadtest

# Custom parameters
https://your-app.azurewebsites.net/api/loadtest?workIterations=500&bufferSizeKb=10000

# Thread pool only (no CPU work)
https://your-app.azurewebsites.net/api/loadtest?workIterations=0&bufferSizeKb=100

POST con el Cuerpo Solicitado

Para escenarios más complejos, usa POST con un cuerpo JSON :

POST /api/loadtest
Content-Type: application/json

{
  "workIterations": 200,
  "bufferSizeKb": 20000,
  "softLimit": 25,
  "degradationFactor": 500,
  "baselineDelayMs": 500,
  "errorAfter": 120,
  "errorPercent": 20
}
  1. Configurar el patrón de carga (subir en rampa, estar en estado estable, reducir la rampa)
  2. Configurar Azure integración con Monitor para correlacionar métricas

📋 Referencia de parámetros de consulta

Cada parámetro controla un recurso específico de forma independiente:

Parámetro Default Recurso objetivo Descripción
workIterations 200 CPU Iteraciones hash SHA256 por ciclo de trabajo CPU . Controla el uso sostenido CPU (~50% por hilo).
bufferSizeKb 20000 Memoria La memoria se mantenía durante toda la duración de la solicitud en la base de datos (20 MB por defecto). Crea una presión de memoria sostenida.
baselineDelayMs 500 Pool de Hilos Duración mínima de la solicitud (ms). CPU trabajo y sueño intercalados durante todo el tiempo.
softLimit 25 Pool de Hilos Solicitudes concurrentes antes de la degradación. Menor = agotamiento más rápido.
degradationFactor 500 Pool de Hilos Retraso adicional (ms) por solicitud sobre softLimit.
errorAfter 120 Caos Segundos antes de que se puedan lanzar errores aleatorios. Configura a 0 para desactivar.
errorPercent 20 Caos Probabilidad porcentual (0-100) de lanzar un error aleatorio por intervalo de comprobación después del umbral.

📊 Entendiendo el límite blando

El límite suave crea una curva de degradación donde los tiempos de respuesta aumentan a medida que Las solicitudes concurrentes superan el umbral. La fórmula de degradación es: 

additionalDelay = max(0, concurrent - softLimit) × degradationFactor

Nota: Los tiempos de respuesta reales variarán según tu nivel Azure App Service y actual carga, y otros factores. Utilizar Azure pruebas de carga para establecer el rendimiento de referencia Para tu entorno específico. Los parámetros por defecto están ajustados para un trabajador Premium0V3. Cualquier otro tamaño de SKU o operador necesitará ajuste individual.

Ejemplo de patrón de degradación

Con los ajustes por defecto (softLimit=25, degradationFactor=500):

Solicitudes simultáneas Retraso adicional Tiempo de respuesta esperado
10 0ms ~500ms (solo línea base)
25 0ms ~500ms (en límite suave)
50 12.500 ms ~13
100 37.500 ms ~38s

💥 Inyección de errores

El endpoint de la prueba de carga puede inyectar excepciones aleatorias para simular un resultado realista fallos en las aplicaciones. Esto se controla mediante dos parámetros:

  • errorAfter (por defecto: 120) - Segundos antes de que puedan empezar los errores
  • errorPercent (por defecto: 20) - Probabilidad de error una vez alcanzado el umbral

Tipos de excepción

Las excepciones aleatorias se seleccionan de un conjunto de excepciones realistas .NET :

Tipo de excepción Mensaje de ejemplo
InvalidOperationException La operación no es válida debido al estado actual
ArgumentException El valor no se encuentra dentro del rango esperado
NullReferenceException Referencia de objeto no establecida como una instancia de un objeto
KeyNotFoundException La clave dada no estaba presente en el diccionario
TimeoutException La operación ha terminado
HttpRequestException Se produjo un error al enviar la solicitud
OutOfMemoryException Memoria insuficiente para continuar la ejecución
IOException No se pueden leer datos de la conexión de transporte

Desactivar la inyección de errores

# Disable errors entirely
/api/loadtest?errorAfter=0

# Or set errorPercent to 0
/api/loadtest?errorPercent=0

🔍 Diagnóstico de resultados de pruebas de carga

1. Application Insights Integración

  • Ir a Application Insights > Actuación
  • Filtrar por nombre de operación GET /api/loadtest
  • Observar los percentiles del tiempo de respuesta (p50, p95, p99)
  • Comprueba las tasas de fallo para el umbral de error aleatorio configurable (por defecto: 120s)

2. Azure Monitorizar métricas

  • Solicitudes - Número total de solicitudes y tasa
  • Tiempo de respuesta - Latencias medias y percentiles
  • Errores en el servidor HTTP - Errores 5xx por excepciones aleatorias
  • CPU Porcentaje - Debe aumentar con la carga
  • Conjunto de Trabajo de Memoria - Debería aumentar con bufferSizeKb

3. Endpoint de estadísticas de pruebas de carga

Consulta el endpoint de estadística durante la prueba:

GET /api/loadtest/stats

{
  "currentConcurrentRequests": 127,
  "totalRequestsProcessed": 45230,
  "totalExceptionsThrown": 23,
  "averageResponseTimeMs": 847.5
}

📤 Formato de respuesta

Cada solicitud exitosa devuelve:

{
  "elapsedMs": 1247,
  "concurrentRequestsAtStart": 42,
  "degradationDelayAppliedMs": 8500,
  "workIterationsCompleted": 200,
  "memoryAllocatedBytes": 20480000,
  "workCompleted": true,
  "message": "Load test completed in 1247ms"
}

Campos de respuesta

Campo Descripción
elapsedMs Duración total de la solicitud en milisegundos
concurrentRequestsAtStart Número de solicitudes simultáneas cuando comenzó esta petición
degradationDelayAppliedMs Se aplica un retraso adicional debido a exceder el límite blando
workIterationsCompleted Número de iteraciones de hash SHA256 completadas
memoryAllocatedBytes Tamaño del búfer de memoria asignado para esta solicitud
workCompleted Si todo el trabajo se completó con éxito
message Mensaje de completación legible por humanos

✅ Mejores prácticas

HACERLO:

  • ✅ Empieza con parámetros por defecto para establecer una línea base
  • ✅ Monitorizar métricas Azure durante las pruebas de carga
  • ✅ Utiliza el endpoint /api/loadtest/stats para rastrear la concurrencia
  • ✅ Aumenta gradualmente la carga para encontrar puntos de ruptura
  • ✅ Correlaciona los resultados de las pruebas de carga con Application Insights
  • ✅ Prueba con patrones de carga realistas (aumento de carga, estado estable, reducción de rampa)

NO LO HAGAS:

  • ❌ Empieza con valores extremos de parámetros
  • ❌ Ignora las métricas de memoria cuando uses un bufferSizeKB grande
  • ❌ Ejecuta pruebas extendidas con errorAfter=0 si quieres ver fallos realistas
  • ❌ Supongamos que los resultados de una SKU se aplican a otra
