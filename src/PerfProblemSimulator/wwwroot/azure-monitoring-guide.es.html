☁️ Azure Guía de diagnóstico

Aprende a diagnosticar problemas de rendimiento usando herramientas de diagnóstico Azure App Service

📊 Resumen

Esta guía explica cómo utilizar Azure herramientas de diagnóstico para diagnosticar los problemas de rendimiento generados por el Simulador de Problemas de Rendimiento.

El simulador genera cuatro tipos de problemas:

  1. Alto CPU - Bucles de espín paralelos que consumen todos los núcleos
  2. Presión de memoria - Asignaciones fijadas en el Gran Montón de Objetos
  3. Escaseza de pozas de hilo - Patrones de bloqueo sincronizados sobre asíncronos
  4. Solicitudes lentas - Solicitudes de larga duración para análisis de CLR Perfiladores

Cada problema puede diagnosticarse utilizando herramientas y técnicas Azure específicas.

🔗 Correlación ID de simulación

A cada instancia de simulación se le asigna una única ID de simulación (GUID) que permite una correlación extrema a extrema entre la interfaz del panel de control y los diagnósticos Azure .

Encontrar identificadores de simulación

En el panel de control: Pasa el cursor sobre cualquier mensaje relacionado con la simulación en el Registro de Eventos (marcado con un subrayado punteado) para ver el ID completo de la simulación en una descripción emergente.

En Azure: El mismo ID de simulación se emite en los registros del lado del servidor y puede encontrarse en:

  • Application Insights - Trazas de búsqueda y eventos personalizados por ID de simulación
  • Análisis de registros - Registros de consulta usando el ID como filtro
  • Contenedor/App Service troncos - Filtrar stdout/stderr por el GUID

Flujo de trabajo de resolución de problemas

  1. Inicia una simulación desde el panel de control
  2. Pasa el cursor sobre el mensaje de inicio en el Registro de Eventos para copiar el ID de Simulación
  3. Portal abierto Azure → Application Insights → Registros (o Análisis de Registros)
  4. Consulta los registros usando el ID de simulación para ver toda la actividad relacionada del lado del servidor
  5. Correlacionar las marcas de tiempo entre eventos del panel de control y métricas del servidor

Propina: El ID de Simulación ayuda a aislar la telemetría al ejecutar múltiples simulaciones simultáneas o cuando el panel de control es utilizado por varios usuarios.

🔥 Diagnóstico de CPU

Síntomas

  • La aplicación deja de responder
  • Aumento de los tiempos de espera de las solicitudes
  • CPU métrica muestra un uso sostenido alto (>80%)

Azure Herramientas para Utilizar

1. App Service Diagnosticar y Resolver Problemas

  1. Navega hasta tu App Service en el Portal Azure
  2. Clic Diagnosticar y resolver problemas En el menú izquierdo
  3. Seleccionar Disponibilidad y rendimiento > Análisis de CPU
  4. Revisa el calendario y el desglose del proceso

2. Application Insights Métricas en Tiempo Real

  1. Abre Application Insights para tu aplicación
  2. Ir a Métricas en vivo
  3. Observa el grafo CPU en tiempo real
  4. Observa la correlación con las solicitudes entrantes

3. CPU Perfilado (Avanzado)

  1. En App Service, ve a Diagnosticar y resolver problemas
  2. Busca "CPU Perfilado"
  3. Clic Recoger Rastreo de perfilador
  4. Activa la simulación de CPU
  5. Espera a que termine la recogida
  6. Descarga y analiza el .diagsession archivo

Qué buscar

En el rastreo del perfilador, verás:- Hilos dedicados ejecutando un bucle while - Alto tiempo de autonomía en CpuStressService - Nombres de hilos como CpuStress-*

📊 Diagnóstico de la presión de memoria

Síntomas

  • Aumento del uso de memoria con el tiempo
  • GC frecuencia de recogida aumenta
  • La aplicación puede volverse más lenta debido a GC pausas
  • Eventualmente, OutOfMemoryException

Azure Herramientas para Utilizar

1. App Service Métricas

  1. Navega hacia tu App Service
  2. Ir a Métricas
  3. Añadir métricas: Conjunto de trabajo de memoria, Bytes privados
  4. Ajusta la granularidad del tiempo a 1 minuto
  5. Observa el crecimiento de la memoria durante la simulación

2. Análisis de volcado de memoria

  1. Ir a Diagnosticar y resolver problemas
  2. Busca "Memory Dump"
  3. Clic Recoger Memoria Dump
  4. Elige "Volcado completo" para un análisis detallado
  5. Descárgalo y ábrelo en Visual Studio o WinDbg

3. Application Insights

  1. Abierto Application Insights
  2. Ir a Rendimiento > Dependencias
  3. Busca patrones lentos relacionados con GC

Qué buscar

En un volcado de memoria, busca:- Matrices de bytes grandes[] en el Gran Montón de Objetos (LOH) - GCHandle.Alloc objetos clavados - Objetos que no se recogen debido a la fijación

Patrón de código (Qué pasa)

Este es el código "malo" intencionado en MemoryPressureService: var datos = Nuevo Byte[tamañoMegabytes* 1024 * 1024];var pinnedHandle = GCHandle.Alloc(data, GCHandleType.Pinned);La asignación fijada: // 1. 🧵 Diagnóstico de la falta de pool de hilos

Síntomas

  • Las solicitudes se ponen en cola y se agotan
  • La latencia aumenta incluso para peticiones simples
  • Avisos de agotamiento de la piscina de rosca en los registros
  • Los chequeos médicos pueden fallar

Azure Herramientas para Utilizar

1. Application Insights Dependencias

  1. Abierto Application Insights
  2. Ir a Mapa de aplicaciones
  3. Busca alta latencia en las llamadas salientes
  4. Fíjate en el patrón de "espera hilo"

2. App Service Diagnóstico

  1. Ir a Diagnosticar y resolver problemas
  2. Buscar "Threading"
  3. Revisar el número de hilos y las métricas de contención

3. Contadores de rendimiento

Vigila estos contadores:

  • .NET CLR Thread Pool / # of Threads
  • .NET CLR Thread Pool / Queue Length
  • Process / Thread Count

Qué buscar

Señales de falta de charcos de hilo:- El número de hilos aumenta de forma constante - Longitud de la cola que crece - Largas duraciones de las solicitudes - Tiempos de espera en operaciones simples

🐌 Diagnosticar solicitudes lentas con CLR Profiler

El Simulador de Peticiones Lentas genera peticiones de larga duración (~25 segundos cada una) utilizando patrones sync-over-async diseñados específicamente para ser fácilmente identificables en CLR trazas del perfilador.

Propósito

Esta simulación está diseñada para Formar a los desarrolladores para identificar llamadas bloqueantes en rastros de perfiladores. A diferencia de la simulación de Starvation del Pool de Hilos (que causa muchos bloques cortos), esta crea solicitudes largas individuales que son claramente visibles en las pilas de llamadas.

Azure Herramientas para Utilizar

1. Azure Curación Automática con Disparador de Petición Lenta

  1. Navega hasta tu App Service en el Portal Azure
  2. Ir a Diagnosticar y resolver problemas
  3. Busca "Auto-Curación"
  4. Configurar el disparador de peticiones lentas (por ejemplo, peticiones > 20 segundos)

2. CLR Perfilador / Azure Perfilador

  1. Activa primero la simulación de peticiones lentas — esto priva el pool de hilos, lo que ralentiza la tasa de la sonda de salud y reduce el ruido en la traza
  2. En App Service, ve a Diagnosticar y resolver problemas
  3. Busca "Profiler"
  4. Inicia una CPU Perfilador Trace (recomendados 60-120 segundos)
  5. Descarga y analiza la pista

Qué buscar en CLR Profiler

El servicio utiliza tres patrones de sync-over-async diferentes con nombres de métodos intencionadamente descriptivos:

Escenario 1: Bloqueo simple

Busca métodos: FetchDataSync_BLOCKING_HERE, ProcessDataSync_BLOCKING_HERE, SaveDataSync_BLOCKING_HERE

En el perfilador, verás el tiempo dedicado a: → SlowRequestService.FetchDataSync_BLOCKING_HERE → Thread.Sleep

Escenario 2: Métodos de sincronización anidados

Busca métodos que terminen en: _BLOCKS_INTERNALLY

Estos son métodos síncronos que bloquean internamente usando Thread.Sleep: → ValidateOrderSync_BLOCKS_INTERNALLY → CheckInventorySync_BLOCKS_INTERNALLY → ProcessPaymentSync_BLOCKS_INTERNALLY → SendConfirmationSync_BLOCKS_INTERNALLY

Escenario 3: Base de datos/Patrón deHTTP

Busca métodos que terminen en: Sync_SYNC_BLOCK

Bases de datos simuladas y HTTP bloqueando llamadas: → GetCustomerFromDatabaseSync_SYNC_BLOCK → GetOrderHistoryFromDatabaseSync_SYNC_BLOCK → CheckInventoryServiceSync_SYNC_BLOCK → GetRecommendationsFromMLServiceSync_SYNC_BLOCK → BuildResponseSync_SYNC_BLOCK

💡 Información clave: Tiempo de cola frente a tiempo de ejecución

Si ves un gran hueco donde aparece "No se emiten eventos" antes de que empiece la solicitud:

  • Tiempo de espera: El Pool de Hilos está vacío, así que tu solicitud se quedó en la cola esperando un hilo (por ejemplo, 20+ segundos).
  • Tiempo de ejecución: Una vez asignado un hilo, el método finalmente se ejecutaba.

Ejemplo: El usuario espera 47 segundos, pero el registro muestra solo 25 segundos de duración. ¡Los 22 que faltaban se gastaron en la cola!

💡 Latidos de ETW

El simulador registra "Generando ruido de traza..." cada 500 ms para vaciar los búferes ETW, asegurando que las peticiones largas no se pierdan en las trazas.

💥 Diagnóstico de fallos de aplicaciones

Síntomas

  • La solicitud de repente deja de estar disponible
  • HTTP errores 502/503 devueltos a los clientes
  • Azure reinicia automáticamente la aplicación
  • Los registros de eventos muestran la terminación del proceso

Azure Monitorización de fallos

Azure App Service incluye Monitorización de Fallos integrada que puede capturar automáticamente los volcados de memoria cuando tu aplicación se bloquea.

Activación de la monitorización de fallos

  1. Navega hasta tu App Service en el Portal Azure
  2. Ir a Diagnosticar y resolver problemas
  3. Busca "Monitorización de fallos"
  4. Habilitar la recogida de volcado accidental

⚠️ Importante: Restricciones de la cuenta de almacenamiento

En algunos entornos Azure , la monitorización de fallos puede no funcionar debido a restricciones de seguridad en las cuentas de almacenamiento. Si no puedes usar Monitoreamiento de Fallos, puedes recoger manualmente los volcados de memoria usando ProcDump desde la consola Kudu.

Uso de ProcDump para la recogida manual de volcado

ProcDump es una utilidad de línea de comandos de Sysinternals que puede monitorizar aplicaciones y generar volcados de fallo. Está disponible en Azure App Service a través de la consola Kudu.

Acceso a la consola Kudu

  1. Navega hacia tu App Service
  2. Ir a Herramientas de desarrollo > Herramientas avanzadas
  3. Clic Ve abrir Kudu
  4. Seleccionar Consola de depuración > CMD o PowerShell

Comandos comunes de ProcDump

# Escribe un volcado completo de memoria de un proceso por nombreprocdump -ma w3wp.exe# Escribe un volcado completo cuando ocurre una excepción no gestionadaprocdump -ma -e w3wp.exe# Escribe un volcado completo sobre la excepción de primera o segunda oportunidadprocdump -ma -e 1 w3wp.exe# Escribe hasta 10 volcados, uno por excepciónprocdump -ma -n 10 -e 1 w3wp.exe# Escribe un volcado cuando CPU supere el 80% durante 10 segundosprocdump -ma -c 80 -s 10 w3wp.exe# Escribe un volcado cuando la memoria supere 1GBprocdump -ma -m 1024 w3wp.exe# Escribe un volcado cuando detectan una ventana colgadaprocdump -ma -h w3wp.exe

Referencia de opciones de ProcDump

Opción Descripción
-ma Volcado completo (toda la memoria)
-mm Mini volcado (por defecto, tamaño más pequeño)
-mp Dump del MiniPlus (detallado pero un 10-75% más pequeño que el completo)
-e Volcado en una excepción no gestionada (añadir 1 para la primera oportunidad)
-c CPU porcentaje umbral
-m Umbral de compromiso de memoria en MB
-n Número de volcados por escribir antes de salir
-s Segundos consecutivos antes del volcado (por defecto 10)
-h Vaciar en la ventana colgada
-t Volcado al terminar el proceso

Descargando el archivo de volcado

  1. En Kudu, navega hasta la carpeta donde se creó el volcado
  2. Haz clic en el icono de descarga junto a .dmp archivo
  3. Abre el volcado en Visual Studio, WinDbg u otro depurador

❌ Diagnóstico de solicitudes fallidas (HTTP 5xx)

HTTP 500 errores son uno de los problemas más comunes que afectan a la disponibilidad de aplicaciones. Esta simulación genera excepciones aleatorias para practicar el diagnóstico de solicitudes fallidas en Azure.

Qué simular

  1. En el panel de control, haz clic ❌ Iniciar solicitudes fallidas
  2. Esperar a que se completen las solicitudes (cada una tarda ~1,5 segundos)
  3. Observar los tipos de excepciones en el Registro de Eventos (entradas marrones)

Diagnóstico de AppLens

  1. En Azure Portal, ve a tu App Service
  2. Navegar a Diagnosticar y resolver problemas
  3. Búsqueda de "Errores de aplicación" o "HTTP 500"
  4. AppLens mostrará:
    • Tasa de error a lo largo del tiempo
    • Tipos de excepción más comunes
    • Trazas de pila para cada fallo
    • Correlación con otros síntomas

Diagnóstico de Application Insights

  1. En Azure Portal, ve a tu recurso Application Insights
  2. Clic Fracasos En el menú izquierdo
  3. Revisa la cronología de fallos y el desglose de las excepciones
  4. Haz clic en excepciones individuales para ver:
    • Traza de pila de excepciones completas
    • Detalles de la solicitud (URL, duración, código de respuesta)
    • Propiedades personalizadas y telemetría

Tipos de excepciones comunes que verás

Excepción Causa real Cómo arreglar
NullReferenceException Acceso a la propiedad del objeto nulo Añadir comprobaciones nulas, usar tipos de referencia anulables
TimeoutException Dependencia externa demasiado lenta Aumenta el tiempo de espera, añade un interruptor automático
InvalidOperationException Objeto en estado incorrecto para la operación Revisa las precondiciones, valida el estado
HttpRequestException Llamada de API externa fallida Añadir lógica de reintentos, comprobar la conectividad
KeyNotFoundException Falta una clave de configuración o caché Usa TryGetValue, comprueba primero la existencia

💡 Consejo profesional: Crear alertas de tasa de error

Configura una alerta de Azure Monitor para avisarte cuando la tasa de error de HTTP 5xx supere un umbral:

  1. Ve a tu App Service → Alertas → Crear regla de alerta
  2. Señal: Errores en el servidor HTTP
  3. Umbral: Dinámico o estático (por ejemplo, > 5 por minuto)
  4. Acción: Notificación por correo electrónico, SMS o webhook
