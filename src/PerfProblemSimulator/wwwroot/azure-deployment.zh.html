🚀 Azure 部署指南

部署性能问题模拟器以使用带有OIDC认证的GitHub动作Azure App Service

概述

本指南将带您了解如何使用GitHub Actions with OpenID Connect（OIDC）认证部署性能问题模拟器以Azure App Service。OIDC消除了将凭证作为秘密存储的需求。

前提条件

  • Azure 订阅，允许创建资源
  • Azure AD/Entra ID 创建应用注册权限
  • GitHub 账号
  • Azure CLI安装（可选，适用于CLI方法）

你将创造的东西

资源目的
Azure App Service托管性能问题模拟器应用程序
Azure AD 应用注册GitHub动作的身份OIDC
联邦证书GitHub仓库链接至Azure AD
角色分配授予部署权限

1 创造 Azure App Service

选项A： Azure 门户

  1. 导航至应用服务
    • 前往 Azure 门户
    • 搜索“App Services”并选择它
    • 咔哒 + 创作
  2. 配置基础
    背景设定价值
    订阅您的订阅
    资源组创建新内容或使用现有内容
    名称your-app-name （必须是全局唯一的）
    发布代码
    运行时栈选择你的应用程序运行时间
    操作系统Windows或Linux（根据你的运行时情况）
    地区你偏好的地区
  3. 配置 App Service 计划
    背景设定推荐
    SKU基础B1 或更高

    ⚠️ 重要提示： 免费（F1）层不支持WebSockets，而WebSockets是实时 SignalR 仪表盘更新所必需的。

  4. 审核与创作 - 点击创建资源

选项B：使用命令行Azure

# Login to Azure
az login

# Set variables
RESOURCE_GROUP="your-resource-group"
APP_NAME="your-app-name"
LOCATION="eastus"
APP_SERVICE_PLAN="your-app-plan"
RUNTIME="your-runtime"  # e.g., "dotnet:8", "node:20-lts", "python:3.11"

# Create resource group
az group create --name $RESOURCE_GROUP --location $LOCATION

# Create App Service Plan (B1 for WebSocket support)
az appservice plan create \
  --name $APP_SERVICE_PLAN \
  --resource-group $RESOURCE_GROUP \
  --location $LOCATION \
  --sku B1

# Create Web App with your runtime
az webapp create \
  --name $APP_NAME \
  --resource-group $RESOURCE_GROUP \
  --plan $APP_SERVICE_PLAN \
  --runtime $RUNTIME

启用WebSockets

创建后，启用 WebSockets 以实现 SignalR 实时通信：

  1. 去你的 App Service → 配置 → 一般设定
  2. 赛场 网页套接字 到 关于
  3. 咔哒 保存

或者通过CLI：

az webapp config set \
  --name $APP_NAME \
  --resource-group $RESOURCE_GROUP \
  --web-sockets-enabled true

2 创建 Azure AD 应用注册

GitHub Actions 使用 OpenID Connect（OIDC）与 Azure 进行身份验证，无需存储凭据。

创建应用注册

  1. 导航至应用注册
    • 前往 Azure 门户
    • 搜索“应用注册”并选择它
    • 咔哒 + 新注册
  2. 配置注册
    背景设定价值
    名称github-your-app-deploy （你部署身份的描述性名称）
    支持的账户类型仅限本组织目录中的账户
    重定向URI空着
  3. 点击注册

记录身份证

来自 概述 页面，复制这些值（你需要它们来做 GitHub 秘密）：

价值GitHub 秘密名字
应用（客户端）IDAZURE_CLIENT_ID
目录（租户）IDAZURE_TENANT_ID

获取订阅ID

  1. 前往 订阅 在 Azure 门户中
  2. 选择您的订阅
  3. 复制 订阅ID → 这是 AZURE_SUBSCRIPTION_ID

3 配置联合凭据

联合凭据允许 GitHub Actions 在不存储秘密的情况下进行认证。

  1. 进入您的应用注册页面
    • 前往 证书与秘密
    • 精选 联邦资历 标签页
    • 咔哒 + 添加凭证
  2. 配置凭证
    背景设定价值
    联合凭证情景GitHub 部署 Azure 资源的行动
    组织结构你的 GitHub 用户名或组织
    存储库你的仓库名称
    实体类型分支
    GitHub 分支名称main
    名称github-main-branch
  3. 点击添加

💡 注： 如果你也想从拉取请求或其他分支部署，可以添加带有相应实体类型的联邦凭证。

4 授予 Azure 权限

应用注册需要权限才能部署到你的 App Service。

分配贡献者角色

  1. 导航到你的 App Service
    • 前往 访问控制（IAM）
    • 咔哒 + 加法 → 添加角色分配
  2. 配置角色分配
    背景设定价值
    职责撰稿人
    分配访问权限用户、组或服务主体
    成员搜索您的应用注册名称
  3. 点击审核 + 分配

5 配置 GitHub 秘密

克隆仓库

  1. 克隆仓库
    • 访问你的技术栈的性能问题模拟器仓库
    • 将仓库克隆到你的本地机器

添加 GitHub 秘密

  1. 导航到仓库设置
    • 访问你的仓库 GitHub
    • 咔哒 背景设定 → 秘密与变量 → 行动
    • 咔哒 新仓库秘密
  2. 添加以下秘密
    秘密名字价值描述
    AZURE_CLIENT_IDxxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx应用注册客户端ID
    AZURE_TENANT_IDxxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxxAzure AD 目录/租户 ID
    AZURE_SUBSCRIPTION_IDxxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxxAzure 订阅ID

💡 无需客户秘密：OIDC联合凭证消除了存储客户端秘密的需求。上述三个身份证就足够了。

更新工作流程（如有需要）

更新 .github/workflows/deploy.yml 附上你的 App Service 名称和运行时版本：

env:
  AZURE_WEBAPP_NAME: your-app-service-name  # Your App Service name
  # Add any runtime-specific version variables as needed

6 部署

自动部署

部署会自动触发，当你推入 main 分支：

git add .
git commit -m "Your changes"
git push origin main

手动部署

  1. 访问你的仓库 GitHub
  2. 咔哒 行动 标签页
  3. 精选 部署到 Azure App Service
  4. 咔哒 运行工作流程 → 运行工作流程

验证部署

  1. 打开应用网址: https://<your-app-name>.azurewebsites.net
  2. Verify Dashboard：实时指标应通过 SignalR更新，状态应显示“已连接”
  3. 测试健康终点: 
    curl https://<your-app-name>.azurewebsites.net/api/health

可选配置

你可以用 Azure App Service 环境变量自定义应用行为。

健康探针率

控制服务器发送健康探针的频率以测量请求延迟。所有探针都经过 Azure 前端，以捕捉真实的端到端延迟。

环境变量

变量名称 描述 默认
HEALTH_PROBE_RATE 探测间隔以毫秒计。至少100毫秒。 200

通过 Azure CLI设置

# Slow down probes if CLR profiler shows overlapping requests
az webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \
  --settings HEALTH_PROBE_RATE=400

闲置暂停

当应用程序处于空闲状态（无仪表盘连接或负载测试请求）时，健康探针会自动暂停，以减少不必要的网络流量 Azure前端和 Application Insights 遥测数据。

环境变量

变量名称 描述 默认
IDLE_TIMEOUT_MINUTES 暂停健康探针前的几分钟不活动。当仪表盘打开或收到任何请求时，活动会自动恢复。 20

通过 Azure CLI设置

# Extend idle timeout to 30 minutes
az webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \
  --settings IDLE_TIMEOUT_MINUTES=30

自定义页面页脚

设置一个自定义页脚消息，显示在所有页面上。页脚支持 HTML，允许你包含链接。

环境变量

变量名称 描述
PAGE_FOOTER 页脚第二行的HTML内容。如果没有设置，只显示应用描述和构建信息。

示例价值

Created by <a href="https://yoursite.com">Your Team</a> for training purposes

通过 Azure CLI设置

# Set the footer via Azure CLI
az webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \
  --settings 'PAGE_FOOTER=Created by <a href="https://yoursite.com">Your Team</a> for training purposes'

通过 Azure 门户设定

  1. 在Azure门户中导航到你的App Service
  2. 前往 背景设定 → 环境变量
  3. 咔哒 + 加法
  4. 套装名称： PAGE_FOOTER
  5. 设置值：你的HTML页脚内容
  6. 咔哒 申请，则 确认

GitHub 仓库链接

在侧边栏菜单中显示指向 GitHub 仓库的链接。该链接默认是隐藏的，只有在两个环境变量都配置好后才会出现。

环境变量

变量名称 描述
GITHUB_USER_NAME GitHub 用户或组织名称（例如， contoso)
GITHUB_REPO_NAME 存储库名称（例如， perf-simulator)

通过 Azure CLI设置

# Enable GitHub link in the sidebar menu
az webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \
  --settings GITHUB_USER_NAME=contoso GITHUB_REPO_NAME=perf-simulator

故障排除

SignalR 连接失败

症状： 仪表盘显示“断开连接”，图表不会实时更新

解决方案：

  • 确保 App Service 配置中启用了WebSockets
  • 从免费等级升级到基础级或更高等级
  • 检查ARR亲和力是否被启用（设置→配置→通用设置）

OIDC 认证失败

症状：GitHub 操作在“登录到Azure”步骤失败

检查：

  • 确认三个秘密都设置正确
  • 确认联邦凭证主题与你的仓库/分支机构匹配： repo:USERNAME/REPO:ref:refs/heads/main
  • 确保应用注册在App Service中扮演贡献者角色

启动时应用崩溃

症状： 应用显示“应用错误”或没有响应

检查：

  • 在 Azure 门户中查看日志流（App Service →日志流）
  • 在 App Service → App Service 日志中启用应用日志
  • 验证运行时版本是否符合你的应用需求
  • 检查缺少配置或环境变量

查看日志

# Stream live logs
az webapp log tail --name $APP_NAME --resource-group $RESOURCE_GROUP

# View deployment logs
az webapp log deployment show --name $APP_NAME --resource-group $RESOURCE_GROUP

快速参考 - CLI 一行

完整的设置脚本（顶部可自定义变量）：

# Full setup (customize variables at top)
RESOURCE_GROUP="your-resource-group"
APP_NAME="your-app-name"
LOCATION="eastus"
RUNTIME="your-runtime"  # e.g., "dotnet:8", "node:20-lts", "python:3.11"
GITHUB_REPO="YOUR_USERNAME/YOUR_REPO"

# Create resources
az group create -n $RESOURCE_GROUP -l $LOCATION
az appservice plan create -n "${APP_NAME}-plan" -g $RESOURCE_GROUP -l $LOCATION --sku B1
az webapp create -n $APP_NAME -g $RESOURCE_GROUP -p "${APP_NAME}-plan" --runtime $RUNTIME
az webapp config set -n $APP_NAME -g $RESOURCE_GROUP --web-sockets-enabled true

# Create App Registration and get IDs
APP_ID=$(az ad app create --display-name "github-${APP_NAME}-deploy" --query appId -o tsv)
az ad sp create --id $APP_ID
TENANT_ID=$(az account show --query tenantId -o tsv)
SUB_ID=$(az account show --query id -o tsv)

# Create federated credential
az ad app federated-credential create --id $APP_ID --parameters "{
  \"name\": \"github-main-branch\",
  \"issuer\": \"https://token.actions.githubusercontent.com\",
  \"subject\": \"repo:${GITHUB_REPO}:ref:refs/heads/main\",
  \"audiences\": [\"api://AzureADTokenExchange\"]
}"

# Assign permissions
APP_SERVICE_ID=$(az webapp show -n $APP_NAME -g $RESOURCE_GROUP --query id -o tsv)
az role assignment create --assignee $APP_ID --role Contributor --scope $APP_SERVICE_ID

# Output secrets for GitHub
echo "AZURE_CLIENT_ID=$APP_ID"
echo "AZURE_TENANT_ID=$TENANT_ID"
echo "AZURE_SUBSCRIPTION_ID=$SUB_ID"
