🎯 Simulador de Problemas de Desempenho

Uma aplicação Azure App Service educacional que cria intencionalmente problemas de desempenho para fins de aprendizado e demonstração.

Propósito

Esta aplicação foi projetada para ajudar Azure App Service Engenheiros de Suporte e Consultores Técnicos:

📖
Aprenda Como diagnosticar problemas comuns de desempenho em Azure App Service
🔧
Prática Usando Azure ferramentas de diagnóstico e monitoramento
🎭
Demonstrar Anti-padrões de desempenho em um ambiente controlado
👥
Trem Apoiar as equipes na identificação e resolução de problemas de desempenho

⚠️ Aviso - Este aplicativo cria intencionalmente problemas de desempenho!

  • 🔥 CPU estresse- Cria threads dedicadas executando loops de spin para consumir todos os núcleos CPU
  • 📊 Pressão de memória - Aloca e fixa blocos de memória para aumentar o conjunto de trabalho
  • 🧵 Falta de poços de fios - Usa sync-over-async anti-padrões para bloquear threads pool de threads
  • 🐌 Pedidos lentos - Gera requisições de longa duração com padrões sync-over-async para análise de CLR Profiler
  • 💥 A aplicação trava - Desencadeia travamentos fatais para testes Azure Monitoramento de Falhas e despejos de memória
  • ❌ Pedidos fracassados - Gera HTTP erros 5xx para testes de AppLens e Application Insights

Implante essa aplicação apenas em ambientes isolados e não produtivos.

📊 Painel

A aplicação inclui um painel em tempo real na URL raiz que mostra:

  • CPU uso - Utilização atual do processador
  • Memória - Tamanhos de conjunto de trabalho e GC heap
  • Pool de fios - Threads ativas e comprimento da fila
  • Latência de solicitação - Tempo de resposta da sonda em tempo real (mostra o impacto da falta de pool de threads)
  • Simulações ativas - Atualmente executando simulações de problemas

O painel utiliza SignalR para atualizações em tempo real e inclui controles para acionar cada tipo de simulação.

Indicadores de Cor Métrica

Os blocos de métricas CPU e Memória usam codificação dinâmica de cores para indicar os níveis de utilização de recursos:

Cor Utilização Status
Preto (padrão) 0-60% Normal - os recursos estão dentro do alcance aceitável
Amarelo 60-80% Aviso - uso elevado, aproximando-se de níveis preocupantes
Vermelho >80% Perigo - alto uso, potencial esgotamento de recursos

Nota: Os limiares de memória são calculados dinamicamente com base na memória total real disponível reportada pelo servidor, não em um valor codificado fixamente. Isso garante avisos precisos, independentemente da configuração da RAM da máquina.

IDs de Registro de Eventos e Simulação

O Registro de Eventos na parte inferior do painel registra toda a atividade da simulação com entradas codificadas por cores que correspondem a cada tipo de simulação. Mensagens relacionadas a uma instância específica de simulação são marcadas com um sublinhado pontilhado.

Visualização e Cópia de IDs de Simulação: Passe o mouse sobre qualquer mensagem sublinhada para revelar a informação completa ID de simulação (GUID) em uma dica de ferramenta. Clique na mensagem para copiar o ID da Simulação para sua prancheta — a mensagem destacará brevemente o verde para confirmar a cópia. Esse ID identifica de forma única cada instância de simulação.

Uso de IDs de Simulação para Solução de Problemas:

  • O mesmo ID de simulação é registrado do lado do servidor e aparece no Azure Application Insights e no Log Analytics
  • Use esse ID para correlacionar eventos do painel com logs de servidores ao analisar problemas de desempenho
  • Filtre Azure logs pelo ID de Simulação para isolar a telemetria de uma execução de simulação específica
  • Veja o Azure Guia de Monitoramento para técnicas detalhadas de correlação

⏱️ Monitor de Latência de Solicitação

O painel inclui um Monitor de Latência de Solicitação Isso demonstra como a falta de pool de threads afeta o processamento de requisições no mundo real. Os valores de latência mostrados incluem tanto tempo de fila quanto tempo de processamento.

Como Funciona

  • Uma thread dedicada em segundo plano (não do pool de threads) investiga continuamente /api/health/probe
  • A latência é medida de ponta a ponta: solicitação enviada → resposta recebida
  • Os resultados são transmitidos via SignalR para o painel em tempo real

O que você vai observar

Cenário Latência (total) Status Explicação
Operação normal < 150ms 🟢 Bom Threads disponíveis para pool de threads
Fome leve 150ms - 1s 🟡 Degradado Requisições enfileiradas aguardando por threads
Fome severa > 1s 🔴 Severo Atraso significativo na fila
Tempo Anos 30 🔴 Crítica Nenhum tópico ficou disponível dentro do tempo limite

Por que isso importa

Durante a falta de pool de threads, métricas de CPU e memória frequentemente parecem normais, mas os usuários experimentam latência severa. O monitor de latência torna esse problema invisível visível — você pode ver os tempos de resposta dispararem de milissegundos para segundos ao acionar a simulação sync-over-async .

🔥 CPU Simulação de Estresse

A simulação de CPU Tensão cria alto uso de CPU por meio de loops paralelos de spin, saturando todos os núcleos de CPU disponíveis por um período especificado.

Como Funciona

A simulação gera um thread por CPU núcleo, cada um executando um ciclo de giro de espera ocupada (incrementando um contador em um restrito while(true) loop). Isso consome 100% da capacidade CPU em todos os núcleos.

Controles do Painel

  • Duração: Quanto tempo fazer o teste de estresse (10-300 segundos, padrão 30)
  • Início/Parada: Iniciar ou cancelar a simulação

O que você vai observar

Métrica Durante a simulação Explicação
CPU % ~100% Todos os núcleos saturados por laços de spin
Memória Inalterado Loops de spin não alocam memória
Latência Pode aumentar Menos CPU disponível para processamento de solicitações

Questões CPU do Mundo Real

Alta CPU na produção é tipicamente causada por:

  • Algoritmos ineficientes (O(n²) quando O(n) é possível)
  • Loops infinitos devido a bugs
  • Retrocesso excessivo de expressões regulares
  • Consultas LINQ não otimizadas com grandes conjuntos de dados
  • Esperando em vez de usar async/await

Ferramentas de Diagnóstico

  • dotnet-counters monitor -p {PID} --counters System.Runtime
  • dotnet-trace collect -p {PID} para CPU perfilamento
  • Azure App Service → Diagnosticar e Resolver Problemas → CPU Uso
  • Application Insights Lâmina de desempenho

💾 Simulação de Pressão de Memória

A simulação de Pressão de Memória aloca e mantém grandes arrays de bytes para demonstrar padrões de consumo de memória e comportamento de coleta de lixo.

Como Funciona

Grandes arrays de bytes (mínimo 10MB, padrão 100MB) são alocados usando GC.AllocateArray<byte>(pinned: true). A alocação fixada impede que o coletor de lixo mova a memória, simulando padrões problemáticos de alocação.

Controles do Painel

  • Tamanho (MB): Quantidade de memória a ser alocada (10-500MB)
  • Alocar: Adiciona um novo bloco de memória
  • Divulgar tudo: Libera todos os blocos e acionamentos alocados GC

O que você vai observar

Métrica Após a Alocação Explicação
Memória (MB) Aumentos pelo tamanho da alocação Os gráficos refletem o aumento do conjunto de trabalho
CPU Breve pico Alocação de memória e zeramento
Após o lançamento Quedas de memória GC. Collect() recupera memória

Grande Pilha de Objetos (LOH)

Objetos maiores que 85KB são alocados no Grande Monte de Objetos:

  • O LOH é coletado com menos frequência (apenas coleções da Geração 2)
  • Alocações fixadas impedem compactação de memória
  • Pode levar à fragmentação da memória ao longo do tempo
  • Nossas alocações (10MB+) sempre vão para o LOH

Problemas de Memória no Mundo Real

  • Coleções estáticas que acumulam dados sem limites
  • Gerenciadores de eventos não sendo cancelados
  • Implementação IDisposável inadequada
  • Cache sem limites de tamanho ou expiração
  • Manter referências a objetos grandes por mais tempo do que o necessário

Ferramentas de Diagnóstico

  • dotnet-dump collect -p {PID} para despejos de memória
  • dotnet-gcdump collect -p {PID}para análise de heap GC
  • Visual Studio Memory Profiler
  • Azure App Service → Lâmina do Conjunto de Trabalho de Memória

🧵 Falta de Poças de Fios

A simulação Thread Pool Starvation demonstra o perigoso "sync-over-async" anti-padrão, onde blocos de código síncrono aguardam a conclusão das operações assíncronas.

Como Funciona

A simulação gera múltiplas tarefas paralelas (padrão: 2× CPU núcleos) que cada uma chama Task.Delay().Wait(). Isso bloqueia as threads do pool de threads, causando:

  1. Threads do pool de threads ficam bloqueados na espera
  2. .NET adiciona novos fios aos poucos (~1 por segundo)
  3. Solicitações recebidas ficam em fila aguardando por threads
  4. Latência de requisição dispara dramaticamente

Controles do Painel

  • Duração: Quanto tempo dura cada operação de bloqueio (padrão 15s)
  • Tópicos: Número de fios bloqueadores a serem gerados
  • Início/Parada: Iniciar ou cancelar a simulação

O Anti-Padrão Sincronizado sobre Assíncrono

Esses padrões bloqueiam os fios e devem ser evitados:

// ❌ BAD - Blocks thread waiting for Task
var result = SomeAsyncMethod().Result;
var result = SomeAsyncMethod().GetAwaiter().GetResult();
SomeAsyncMethod().Wait();

// ✅ GOOD - Properly awaits the Task
var result = await SomeAsyncMethod();

O que você vai observar

Métrica Durante a fome Explicação
CPU BAIXO Os tópicos estão esperando, não funcionando
Memória Normal Nenhum vazamento de memória ocorrendo
Contagem de Fios Subida devagar .NET adiciona ~1 fio por segundo
Latência 🔴 Picos para segundos Requisições enfileiradas aguardando por threads

Por que CPU é baixo, mas a latência é alta

Essa é a marca registrada da falta de poço de fios. O CPU não está funcionando porque todas as threads estão bloqueadas esperando. Novos pedidos não podem começar porque não há threads disponíveis. Parece que o app está inativo, mas os usuários experimentam tempos de espera.

Causas do Mundo Real

  • Chamando bibliotecas assíncronas a partir do código de sincronização: httpClient.GetAsync(url).Result
  • Mistura de sincronização e assíncrono em construtores
  • Usando .Result em propriedades
  • Chamando métodos assíncronos a partir do Dispose()
  • Bibliotecas de terceiros que bloqueiam internamente

Como Consertar

  • Uso await Até o topo da pilha de chamadas ("ASSÍNCRONO até o mais embaixo")
  • Evite .Result, .Wait(), e .GetAwaiter().GetResult()
  • Uso ConfigureAwait(false) No código de bibliotecas
  • Considere IAsyncEnumerable para dados em streaming

Ferramentas de Diagnóstico

  • dotnet-counters: Assista a "ThreadPool Thread Count" e "ThreadPool Queue Length"
  • Os tempos de resposta disparam para TODOS os endpoints simultaneamente
  • Azure App Service → Diagnosticar e Resolver Problemas → Contagem de Threads

🐌 Simulador de Solicitação Lenta

O Slow Request Simulator gera requisições de longa duração (~25 segundos cada) usando sync-over-async padrões especificamente projetados para CLR Treinamento de perfilador.

Propósito

Diferente da simulação de Starvation do Pool de Threads (que causa muitos bloqueios curtos), esta cria requisições longas individuais que são claramente visíveis em pilhas de chamadas de perfilador. Os nomes dos métodos são intencionalmente descritivos para facilitar a identificação dos padrões de bloqueio.

Três Cenários de Sincronização sobre Assíncrono

Cenário Padrão de Método O que procurar
Simples _BLOCKING_HERE Task.Delay().Wait() causando bloqueio
Aninhado _BLOCKS_INTERNALLY Métodos de sincronização que bloqueiam internamente
Banco de dados _SYNC_BLOCK GetAwaiter().GetResult() Padrão

Configurações Recomendadas

Para um rastreamento CLR Profiler de 60 segundos:

  • Tempo de Execução do Pedido: 25 segundos (apenas tempo de processamento, excluindo tempo de fila)
  • Intervalo: 10 segundos (múltiplas amostras)
  • Pedidos Máximos: 6 (não sobrecarrega)

Nota sobre o tempo de fila: Se o pool de threads estiver esfomeado, você pode ver uma grande lacuna no rastreio antes do início da solicitação. Isso é "Tempo de Fila" (aguardar uma thread) versus "Tempo de Execução" (rodar o código).

Usando com CLR Profiler

  1. Inicie um rastreamento CPU Profiler (60-120 segundos)
  2. Click 🐌 Iniciar Pedidos Lentos no painel
  3. Espere o rastreamento ser concluído
  4. Baixe e analise - procure pelos nomes dos métodos descritivos

Veja o Azure Guia de Diagnóstico Para instruções detalhadas de análise do perfilador.

💥 Simulação de Colisão

O painel inclui um Simulação de Colisão recurso para testes Azure monitoramento de falhas e coleta de despejo de memória.

⚠️ Aviso - Isso vai ENCERRAR a inscrição!

Simulações de colisão encerram imediatamente o processo. Azure App Service reinicia automaticamente o aplicativo após um travamento. Certifique-se de ter ativado Azure Monitoramento de Falhas antes de acionar um travamento.

Tipos de Colisão Disponíveis

Tipo Descrição Melhores Para
FailFast Chamadas Environment.FailFast() que termina imediatamente com um Relatório de Erro do Windows Testes gerais de colisão (recomendado)
StackOverflow Recursão infinita causando overflow de pilha - não pode ser detectada Prática de análise de traços de pilha
UnhandledException Lança exceção não tratada na thread em segundo plano Educação sobre tratamento de exceções
ViolaçãoDeAcesso Grava em memória inválida (ponteiro nulo) Depuração de código nativo/inseguro
ForaMemória Aloca memória até que o processo trave Análise de despejo de memória

Azure Configuração de Monitoramento de Falhas

  1. No Azure Portal, vá até sua App Service
  2. Navegar para Diagnosticar e resolver problemas
  3. Busca por "Monitoramento de Falhas"
  4. Ative o monitoramento de falhas com "Coletar Despejo de Memória"
  5. Desencadeie um travamento usando o painel ou API
  6. Baixe o dump de travamento do portal Azure
  7. Analise com Visual Studio ou WinDbg

❌ Simulação de Solicitações Falhadas

O painel inclui um Pedidos Fracassados recurso para gerar HTTP erros 5xx para testar Azure ferramentas de diagnóstico como AppLens e Application Insights.

Propósito

Essa simulação gera HTTP 500 respostas de erro de servidor interno que são visíveis em:

  • AppLens - Azure Portal → Diagnosticar e resolver problemas
  • Application Insights - Lâmina de falhas
  • Azure Monitor - Alertas de taxa de erro

Como Funciona

  1. Gera uma thread em segundo plano para gerar requisições
  2. Cada solicitação chama /api/loadtest com probabilidade de erro de 100%
  3. O endpoint do teste de carga gera um tipo de exceção aleatório
  4. Cada requisição leva ~1,5 segundos para visibilidade de latência

Tipos de Exceção

A simulação lança aleatoriamente um desses tipos de exceção por requisição:

Tipo de Exceção Descrição
NullReferenceException Referência de objeto não definida para uma instância
TimeoutException Operação encerrada
InvalidOperationException Operação não válida para o estado atual
ArgumentException Valor fora da faixa esperada
HttpRequestException Erro enviando HTTP solicitação
DivideByZeroException Tentou dividir por zero
KeyNotFoundException Chave não presente no dicionário
IndexOutOfRangeException Índice fora dos limites do array

Registro de Eventos do Painel

Cada solicitação falhada aparece no Registro de Eventos com:

  • Cor marrom - Estilo distinto para visibilidade
  • ❌ Ícone - Indicador de solicitação falhada
  • Tipo de exceção - A exceção específica que foi lançada
  • Latência - Duração da solicitação em milissegundos

Veja o Azure Guia de Diagnóstico Para instruções detalhadas de análise AppLens .

⚙️ Configuração Opcional

Você pode personalizar o comportamento da aplicação usando Azure App Service variáveis de ambiente.

Taxa de Sondagem de Saúde

Controle com que frequência o servidor envia sondas de saúde para medir a latência das requisições. Todas as sondas são roteadas pelo frontend Azure para capturar latência realista de ponta a ponta.

Variável Ambiente

Nome da variável Descrição Padrão
HEALTH_PROBE_RATE Intervalo de sonda em milissegundos. Mínimo 100ms. 200

Configuração via Azure CLI

# Desacelerar as sondas se CLR perfilador mostrar solicitações sobrepostasaz webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \ --configurações HEALTH_PROBE_RATE=400

Tempo de inatividade

Quando a aplicação está ociosa (sem conexões no painel ou requisições de teste de carga), as sondas de saúde são automaticamente suspensas para reduzir o tráfego desnecessário de rede para a interface e a telemetria Application Insights da Azure.

Variável Ambiente

Nome da variável Descrição Padrão
IDLE_TIMEOUT_MINUTES Minutos de inatividade antes de suspender as sondas de saúde. A atividade é retomada automaticamente quando o painel é aberto ou qualquer solicitação é recebida. 20

Configuração via Azure CLI

# Estenda o tempo de espera para 30 minutosaz webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \ --configurações IDLE_TIMEOUT_MINUTES=30

Rodapé de página personalizado

Defina uma mensagem personalizada no rodapé que apareça em todas as páginas. O rodapé suporta HTML, permitindo incluir links.

Variável Ambiente

Nome da variável Descrição
PAGE_FOOTER Conteúdo HTML para a segunda linha do rodapé. Se não estiver definido, apenas a descrição do app e as informações da build são mostradas.

Exemplo de Valor

Criado por <a href="https://yoursite.com">Sua Equipe</a> para fins de treinamento

Configuração via Azure CLI

# Coloque o rodapé via Azure CLIaz webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \ --configurações 'PAGE_FOOTER=Criado por <a href="https://yoursite.com">Sua Equipe</a> para fins de treinamento'

Ambientação via Azure Portal

  1. Navegue até sua App Service no Portal Azure
  2. Vá para Configurações → Variáveis do ambiente
  3. Click + Adicionar
  4. Nome do Conjunto: PAGE_FOOTER
  5. Valor de Definição: Seu conteúdo do rodapé HTML
  6. Click Candidate-se, então Confirme

GitHub Link do Repositório

Exiba um link para o repositório GitHub no menu lateral. O link é oculto por padrão e só aparece quando ambas as variáveis de ambiente estão configuradas.

Variáveis Ambientais

Nome da variável Descrição
GITHUB_USER_NAME GitHub nome de usuário ou organização (por exemplo, contoso)
GITHUB_REPO_NAME Nome do repositório (por exemplo, perf-simulator)

🌐 Seleção de Idioma (Tradução UI)

A interface do painel pode ser exibida em qualquer idioma suportado por Azure Tradutor de Serviços Cognitivos. As traduções são geradas automaticamente no início usando IA e armazenadas em cache para solicitações subsequentes.

⚠️ Aviso de Tradução por IA: As traduções são geradas por Azure Tradutor de Serviços Cognitivos (tradução automática) e podem não ser completamente precisas. Termos técnicos (por exemplo, CPU, SignalR, ThreadPool) são intencionalmente preservados em inglês.

Como Funciona

  1. Defina o UI_LANGUAGE variável de ambiente para um Código da linguagem ISO 639-1 (por exemplo, es para espanhol, fr para o francês, ja para japonês)
  2. Ao iniciar, o app verifica se existe um arquivo de localização traduzido para aquele idioma
  3. Se não (ou se as strings de origem em inglês mudaram), ele chama Azure tradutor para gerar uma
  4. O arquivo traduzido é armazenado em cache — startups subsequentes usam o cache sem custo de tradução
  5. O painel carrega as strings traduzidas e as aplica à interface

Variáveis Ambientais

Nome da variável Descrição Padrão
UI_LANGUAGE Código de idioma ISO 639-1 para a interface do painel. Definir para en para o inglês (sem necessidade de tradução). en
TRANSLATOR_API_KEY Azure chave de assinatura do Tradutor de Serviços Cognitivos. Obrigatório para línguas que não são inglês. (não definido)
TRANSLATOR_ENDPOINT Tradutor API URL do endpoint. https://api.cognitive.microsofttranslator.com
TRANSLATOR_REGION Azure região do recurso Tradutor. eastus

Configuração via Azure CLI

# Definir a interface para espanholaz webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \ --settings UI_LANGUAGE=es TRANSLATOR_API_KEY=sua-chave TRANSLATOR_REGION=eastus

Ambientação via Azure Portal

  1. Navegue até sua App Service no Portal Azure
  2. Vá para Configurações → Variáveis do ambiente
  3. Click + Adicionar e nome do conjunto: UI_LANGUAGE, Valor: seu código de idioma (por exemplo, es)
  4. Click + Adicionar e nome do conjunto: TRANSLATOR_API_KEY, Valor: sua Azure Chave tradutora
  5. Click + Adicionar e nome do conjunto: TRANSLATOR_REGION, Valor: sua região de recurso tradutor
  6. Click Candidate-se, então Confirme (reinício necessário)

Personalizando Termos Protegidos

Termos técnicos que nunca deveriam ser traduzidos estão listados em wwwroot/locales/no-translate.json. Edite este arquivo para adicionar ou remover termos. Os termos são automaticamente preservados em inglês durante a tradução.
